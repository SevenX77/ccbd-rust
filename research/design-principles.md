> **operator 自己要遵守的自律原则**(需求追溯、疗效闭环、干预即补机制、隔离运维、串并行验收)**不在此**,见 `research/operator-discipline.md`。
> 上位文档:`research/perception-layer-first-principles.md`、`perception-final-convergence-2026-07-09.md`。

> 分两部分:**一、通用设计原则**(跨模块普适)+ **二、本项目架构设计原则**(ah 特定,不普适,但在本项目里是硬架构约束)。
> 用法:o1/d1 每过一个设计 gate,逐条对表打钩;违反某条要么改设计,要么显式记例外理由。operator 审设计时同样逐条查。

## 一、通用设计原则

### P1 · 第一性原理,不做补丁思维
每个架构域先写「理想基准」→ 差距表 → 收敛;同族 bug 出现两次 = 结构病,升维重构而非再补一层。
**反例**:给 pane scanner 加幽灵检测/banner 白名单 = 教猜谜的东西更多技巧 = 又一层硬编码。完成检测「停下==完成」把回合边界错译为任务完成,补丁越叠越多。
**检查**:提出一个修法时问——这是消除病根,还是又加一个特判?

### P2 · 模块化,低耦合高内聚
每个模块单一职责、契约清晰;跨模块只经**显式契约**通信,不隐式共享状态。
**正例**:orchestration-substrate 收敛出单一缺失原语 `StateReconciler`(物理↔逻辑收敛的唯一 owner),而非把收敛逻辑散在各处。

### P3 · 不做后兼容
产品重塑期,砍历史包袱优先;有更干净的设计就换掉,不为旧接口/旧数据留 shim。
**配套纪律**:过渡态代码(如 pane-scan fallback)必须标注为可回收 scaffold,不许沉淀成永久层。

### P4 · 效率优先,不做多余动作
为正确/干净可以付大动作成本(worktree/fork/重构),但不做无价值的仪式动作;无效等待(有活在飞却空转)要主动消除。
**反例**:同一 PR 全量串行测试跑多遍;肉眼轮询 CI(该机制化)。

### P5 · 设计冗余原则(生命周期确认)★ 2026-07-12 重点确立
推动生命周期 = 一个 loop(while);跳出 loop 的条件 = **一个明确的、生命周期改变的 100% 可靠信号**。条件==false 时的动作序列**也拆成同构子 loop**,每步有自己的可观测确认。**每一步都要做到 100% 靠谱,否则就是现在这样——永远修不完的 bug。**
**关键澄清(用户亲自厘清)**:
- 100% 可靠的开始信号也只能证「某任务开始了」,**证不了「开始的是我要发的那个 brief」**。composer 残留、Enter 被吞、brief 没贴进去就按 Enter → 内容错发或永久傻等。
- 内容绑定**只能靠动作层逐步核验**补上(清框→验空→贴→验内容一致→Enter→验发出),这与「hook 作生命周期状态权威」不冲突——两码事:动作层管「我发对了没」,hook 管「任务真开始了没」。
**反例族**:dispatch-ACK 竞态、幽灵提示词、banner 误闩、R2 探针被自己注入击穿——全是「没有 100% 可靠信号,靠推断」的同一结构病。

### P6 · pane 不做生命周期状态权威(三层切分)★ 2026-07-09 立、07-12 厘清边界
- **状态层【禁】**:pane 文本内容 → 生命周期状态判定(完成/存活/忙)。**永久禁**。状态权威唯一来自 hook + cgroup/pidfd。
- **动作层【必须】**:核验自己刚写入动作的回显(不注入探测字符,只读「我刚做的写动作生效没」)。必需的设计冗余(见 P5)。
- **诊断层【允许】**:UNKNOWN 上抛后 master 亲自看 pane double-check;hook 与 pane 不匹配 → 以 hook 为准触发错误流程。

### P7 · fail-closed,响亮不静默
信号丢失/超预算 → 判 UNKNOWN 响亮上抛,**绝不静默续跑、绝不用弱信号顶班**(弱信号只作告警佐证)。看门狗 = 权威失败判决,不是退回低级信号的理由。STUCK 是死胡同终态,fail-closed 必须落 QUEUED+IDLE 或 CRASHED。
**反例**:leak-gate `git grep` 执行失败被 `| grep -q` 吞掉 → 静默放行(已修:码≥2=执行失败,拒绝静默)。

### P8 · 撤销「停下==完成」,改显式声明
完成必须是 agent 显式声明(hook 报告)+ 物理背书(cgroup populated=0),不是从「停下/回合结束」推断。开始同理,要显式开始 hook,不靠「发了就假定开始」。
**落点**:D 相位双锁 AND(Stop hook 逻辑锁 + cgroup populated=0 物理锁)。

### P9 · 信号权威分层,低层不覆盖高层(原 D3,用户确认这是设计原则)
多信号并存时定死优先级;低置信信号永不覆盖高置信信号造状态,冲突 → 触发错误流程而非「就近取一个」。
**反例**:R2 就绪探针被自己注入击穿、cgroup「有进程在跑」误当开工——都是低层信号越权。

### P10 · 调研做足,考虑全局影响,采行业最佳实践 ★ 2026-07-12 用户新增
设计任何模块功能之前:
1. **调研做足**——先摸清现状、既有契约、上下游依赖,不拍脑袋起草;
2. **考虑对全局(系统级)的影响**——本模块改动如何波及其它模块/整体不变量,不做只顾局部的局部最优;
3. **找行业最佳实践方案**——同类问题业界成熟怎么解(协议、模式、开源实现),优先对齐成熟方案而非自造轮子;偏离最佳实践要有明确理由。
**动机**:多起返工源于「没调研就设计→撞到全局约束→推翻重来」。调研成本远低于返工成本(呼应 P4 效率优先:调研是有价值的大动作,不是仪式)。

## 二、本项目架构设计原则(ah 特定,非通用)

### A1 · job 状态为第一信号,产物 double-check 实锤,job-id 契约串联 ★ 2026-07-12 用户裁决(修正原 P8)
> 本条是**本项目的重要架构设计原则**,不是普适设计原则。信号可靠性在逐步往靠谱推进,主线锚点选定如下:

1. **第一优先级 = 统一看 job 状态**。生命周期的主信号是 job 状态机(由 hook 事件驱动、逐步硬化),不是产物轨。
2. **产物 double-check 是实锤补强,不是主信号**。job 状态定主,产物核验(git HEAD / 落盘文件 / 约定 sentinel)兜底**证伪**。**只看 double-check 也会出问题**(产物没落盘 ≠ 任务没进展;产物落了 ≠ job 语义完成)。两者结合,不偏废。
3. **恢复 job-id 契约(ccb 最初有,被我们扔掉了)**:job 的**发送 → 执行 → 完成**必须由**同一个 job-id 串联**;hook 事件必须携带 **job-id + 状态信号**,不能只发裸状态。这是可追溯、防错配(dispatch-ACK 竞态、假完成配错 job、跨 tick 串状态)的结构基础。
**动机**:当前完成/开始检测的一批同族病,根子之一是「job 生命周期没有贯穿始终的 id 锚」——事件发出去认不回是哪个 job 的哪个阶段,只能靠时间窗/pane 推断,于是错配、假完成、竞态层出不穷。

> 注:历史 memory「真信号=产物轨 git HEAD 非 job 状态」是**特定情境**下的应急做法(某 provider job 状态已知不可靠时锚产物);一般原则以本条为准——**job 状态第一,产物 double-check 补强**。

### A2 · 三层验收模型:本地内环 / CI 门 / 端到端 smoke ★ 2026-07-12 用户批准
> 「绿」本身一文不值,关键是**每层能不能证它该证的东西**。验收分三层,职责不可互相顶替:

1. **本地内环**(实施者自证):`cargo check` + **本次新增的定点单测**(RED→GREEN)。串行 `CARGO_BUILD_JOBS=1`。**不是验收信号,禁全量 `cargo test`**(慢 + 串行掩盖并发 bug + VPS 上并行 OOM 杀活栈)。只给实施者快速反馈。
2. **CI 门**(正确性验收):并行 + 双平台 + 全量。**唯一的代码正确性验收门**。本地绿 ≠ CI 绿(掩盖过 HOME-override bug)。
3. **端到端 smoke / 换血 dogfood**(拓扑&激活验收):用**真二进制**端到端 spawn 真 worker,跑通真实生产拓扑。**抓 CI 单测都抓不到的拓扑/激活型 bug**(current_exe=ahd 事故:测试单体二进制掩蔽了双二进制生产事实,本地和 CI 单测全绿、生产全崩)。这一层过去当「operator 手动兜底」在做,**必须固化成显式 gate**,不能靠人肉。
**反例(现行病)**:current_exe bug——本地绿、CI 单测绿,只有端到端换血才暴露。→ 拓扑型 bug 必须有第 3 层。
**检查**:一个改动声称「完成」时问——它过了哪几层?拓扑/激活相关的,有没有第 3 层端到端背书?

## 用法
- **设计 gate 检查表**:o1/d1 新设计过审,逐条对 P1–P10 + A1 打钩;违反要么改设计要么显式记例外理由。
- **operator 审设计时**同表复查(operator 自己的自律原则另见 `operator-discipline.md`)。
- **活文档**:用户增删原则即追加,回链具体设计/事故出处。
