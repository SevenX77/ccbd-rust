# ccbd-rust 架构深度分析报告 (by Gemini-Deep-v2)

## 1. 核心判断
`ccbd-rust` 的重写并非简单的语言迁移，而是从"脆弱的胶水脚本"向"强一致性状态机"的范式转型。当前 CCB 的核心病灶在于**观测语义与执行语义的脱节**（Liveness != Readiness）以及**分布式竞态下的静默失效**。Rust 的引入提供了物理层面的并发护栏，但系统的生命力将取决于其 **Reconciliation Loop (调谐循环)** 对外部不确定性（PTY 挂死、LLM 进程僵死、文件系统脑裂）的容错处理能力。

## 2. 独立分析与设计判断

### Step A: 基于 48 条 Observations 的分布式系统视角分析

通过对 Evidence Pack 类 2 的分析，我认为 Claude 视角的 `synthesis` 虽然捕捉到了症状，但漏掉了以下分布式系统设计的核心命题：

1.  **"盲投"协议下的原子性缺失 (O-19, O-20, O-21)**:
    - **判断**: 当前通过 `load-buffer -> paste-buffer -> Enter` 的序列是典型的 "Fire and Forget" 模型。在 PTY/Terminal 环境下，`send-keys` 返回成功仅代表字节已写入伪终端缓存，不代表业务进程（Agent CLI）已读取或执行。
    - **根因**: 缺乏应用层的 ACK 机制。
    - **盲点**: Claude 试图通过加 `sleep` 或 "双 Enter" (O-22) 解决，这在分布式系统中是典型的 "Anti-pattern"。
    - **设计建议**: `ccbd-rust` 必须建立基于 `vt100` 内容解析或 `hook` 信号的 **交付确认机制**，而非依赖时间窗口。

2.  **存活探测 (Liveness) 与就绪探测 (Readiness) 的混淆 (O-33)**:
    - **判断**: O-33 记录了 Codex pane 存活但进程死循环。
    - **分布式视角**: 一个健康的节点必须同时满足 "进程存在 (Liveness)" 和 "能处理请求 (Readiness)"。
    - **设计建议**: `ccbd-rust` 的 `agent.status` 不能只查 `/proc` 或 `tmux list-panes`。必须引入 **Active Heartbeat**（例如监控 O-27 提到的 Activity Markers）或周期性的探测指令。

3.  **确定性清理与状态隔离 (O-05, O-14, O-31)**:
    - **判断**: O-05 提到的 `run_id` 清理和 O-31 的 `tmux_cleanup` 是防御性编程的体现。
    - **根因**: 系统中存在大量的 "隐式关联"（如 PID 记录在临时文件）。
    - **设计建议**: 必须实现 **Idempotent Cleanup (幂等清理)**。无论系统何时启动，Reconciliation Loop 都要能通过扫描 SQLite (SoT) 和实际进程树，将孤儿进程强行归位。

### Step B: 基于文档规范的硬约束清单 (Contract)

根据 Evidence Pack 类 1，`ccbd-rust` 在设计 L1 驱动时必须满足以下"硬性契约"：

1.  **Codex 契约**:
    - **事件驱动**: 必须使用 `--json` (codex-evidence-pack.md:20) 获取 newline-delimited JSON 状态变更，禁止解析文本进度条。
    - **断点恢复**: `resume` 必须显式传递 `SESSION_ID` (codex-evidence-pack.md:19)，并处理 ID 过期的 `failed` 状态（对应 O-33）。

2.  **Claude Code 契约**:
    - **拦截机制**: `PreToolUse` Hook (codex-evidence-pack.md:23,25-27) 必须是 **Blocking** 的。`ccbd-rust` 必须支持从 stdin 喂入 JSON Context 并读取退出码（Exit 2 = Error）。
    - **生命周期**: 必须区分 `resume` 启动和 `start` 启动的不同 Entrypoint (codex-evidence-pack.md:24)。

3.  **Gemini CLI 契约**:
    - **流式输出**: 优先使用 `stream-json` (codex-evidence-pack.md:30) 以获得最低延迟的完成信号。
    - **沙盒注入**: 必须在启动前将 `BeforeAgent` Hook 注入 `~/.gemini/settings.json` (对应 O-37)。

### Step E: 风险清单

- **(A) Blocker**: **PTY 交付确认失效 (O-19)**。如果 `ccbd-rust` 不能保证字节被 Agent 进程读取（而非仅仅写入 PTY），系统在高并发下仍会发生 "Message Stuck"。
- **(B) 应解项**: **Provider 启动竞态 (O-29)**。Gemini 等 Agent 启动慢，Hook 触发晚。需要参数化的 `settle_window`。
- **(C) 监控项**: **Sandbox 性能损耗**。bwrap 大量 Bind-mount 在高频 spawn 场景下的开销。
