## 处境事实(只给事实,不给结论——结论是你要产的)

ah 是隔离 agent 编排底座。当前两大结构痛点,本设计轮要出设计答案:

### A. 感知层(多信号如何裁决 agent/job 的生命周期状态)
- 现有多个**独立信号源**在判 agent 是否完成/卡死:OS 层(进程/cgroup)、hook 事件(agent CLI 的 stop/start 上报)、transcript 日志、pane 文本。它们各自观察、**无统一仲裁**;协调仅靠 DB CAS 先到先得 + 零散让位 if。
- 已拍板的运行铁律(不可违反,你的设计不得触碰):**pane 文本永不用于造状态**(只发 alert);不对干活中 agent 注入按键;完成判定要靠 agent 显式信号(hook)或权威验证,不靠 pane 静止推断。
- 已知事故族:信号缺席被静默当成某个强状态(如 pane 静止→判完成、hook 丢失→退回猜测);STUCK 是超时推断出来的死胡同终态。
- hook 上报通道:hook 进程**发完消息即退出**。
- 部署形态:master/daemon 是独立进程,agent 在 tmux pane 里、被 systemd transient scope 记账(cgroup v2 可用)。

### B. 控制面(job 与 agent 的状态管理)
- **job 没有状态机**:约 11 处裸写 `UPDATE jobs SET status=...`,合法性散在各处 WHERE 子句,无单一写权威、无合法转移表。
- **job 完成与"回合结束"硬耦合**:agent 的 turn 结束处理(mark_agent_idle_*)在**同一事务里寄生**地把 job 标完成——job 完成成了回合结束的副产物。这是"假完成"事故族最深的结构根源(回合结束 ≠ 任务真完成)。
- 多写入口、竞态:派发/认领/恢复/取消各自写 job/agent 状态。

## 你要独立发散的设计问题(给你的答案 + 备选 + 失效模式)

**感知层(四道)**:
1. **单写入口的硬约束形态**:如果要求"agent/job 状态变更只有一条入口",怎么把这条约束做成**编译期或审计可机械校验**的硬约束(而非口头约定)?给具体形态(类型系统/模块封装/lint/CI 检查…),含代价与绕过风险。
2. **各信号类的"缺席=Unknown"预算**:hook 静默多久 / 日志静默多久 / OS 信号不确定多久 → 才从"未知"升级为权威判决(告警/失败)?**按信号类分别定**,给你的取值区间 + 依据 + 为什么不同类不同值。
3. **父/子 cgroup 委托布局**:能否把 agent CLI 放父 scope、它 spawn 的 shell/子进程放**委托子 cgroup**,使"子 cgroup 有无存活进程"这一内核信号能干净回答"除 agent 本体外还有没有活儿在跑"?给布局方案 + 可行性 + 若不可行的退路。
4. **hook 上报归属竞态**:hook 进程发完即退,消息可能在被处理前发送者已死→丢失/无法归属。设计一个**不依赖发送者存活**的归属机制(job-cookie/落盘/序列…),使 hook 信号不再偶发丢失且能对号入座。

**控制面(两道)**:
5. **job 状态机**:给 job 一个真状态机的设计——单一写权威、合法转移集、非法转移如何被结构性挡住。它与感知层"单写仲裁"(问题1)是什么关系(同一个写权威?两个?)。
6. **job 完成与回合结束解耦**:怎么把"job 标完成"从"agent 回合结束"的同事务寄生里拆出来,让 job 完成只由真完成信号(问题2/4 的 hook/权威验证)驱动?给解耦后的时序 + 谁在什么信号下标 job 完成。

**跨层**:7. 感知仲裁器与 job 状态机若是两个写权威,它们如何协作不打架?若合一,如何分层?(你若认为该合/该分,论证。)
