| **TDD Red-Green Loop Enforce**<br>(TDD 红绿循环硬化) | 用实际的测试代码对任务 brief 的每一项契约进行物理锁死，防止重构和实现过程中产生 silent 漂移，确保交付物的高可靠性。对于纯逻辑层、系统集成层、E2E 层分级制定测试严格度。 | [TDD 红绿循环 enforce 落地](feedback_tdd_enforce.md) |

## 第 3 节 · 优化点/反复踩的摩擦

| 摩擦 | 影响 | 可能的改进方向 |
|---|---|---|
| **完成检测超时机制硬编码 (MAX_LOG_MONITOR_WAIT)** | 监听器硬编码 300s 超时时间。若 Codex 做深度 code review 或 cargo test 等真实任务超过 5 分钟，系统会主动丢弃可靠的 log-event 信号并退化为脆弱的 UI 兜底；叠加 `health_check.rs` 的 `.or()` 判定 bug 导致真实 pane 敲字活动被旧 marker 阻断，最终误判 `STUCK` 卡死。 | 1. 将 `MAX_LOG_MONITOR_WAIT` 改为动态延期，如只要 pane 还有新输出或 log 在增量追加就自动为时钟续期；<br>2. 修复 `health_check.rs` 的时间戳取值，使用 `max(last_marker_ts, last_output_ts)` 代替 `.or()`，使物理活动可作为判定依据。 |
| **多 Watchdog 缺乏协调与决策竞态** | 当 master 异常死亡时，`master_watch` 亚秒级复活并 reap-preserve 保留了 worker 的沙箱 HOME，而 `session_watch` 在 7 秒的 debounce 窗口后根据 systemd 单元的 inactive 状态误触发了级联清理，将重新挂载的 worker 目录连带删除。 | 统一使用数据库中的 `sessions.status` 作为唯一的真理仲裁源。当监控发现 anchor inactive 时，必须先 status-gate 检查，若仍为 ACTIVE（代表正在 revive 恢复中）则挂起 defer 不准级联清理，只有在 FAILED 等终态才清理。 |
| **测试沙箱缺乏 GC 导致磁盘耗尽** | 历史集成/e2e 测试运行产生的临时沙箱都持久化写入了 `~/.cache/ah/sandboxes`。由于缺乏 GC 清理，随着用例迭代会疯狂积累（积压了 445 个，高达 48GB），最终把磁盘打满到 100%，阻塞所有构建和编译。 | 1. 改变测试沙箱存储策略，直接写入系统的临时目录（如 `/tmp`）并由测试脚本的 teardown 负责物理回收；<br>2. 引入 ah 自身的 GC 子命令（如 `ah janitor --gc`），定时扫描清理 mtime 超过阈值且无活跃 pid 占用的沙箱。 |
| **Realign 机制的删旧建新非原子性** | 在执行 `ah up` 重整状态时，若发生多个 agent realignment，由于删除和重建不是原子的，在成功删除旧 agent 后，后续的重建操作可能会被其它并发处理打断，导致“拓扑悄缩”（agent 从 DB 和 tmux 双重消失）。 | 重构 realign 逻辑，将删除与重建包装为单个原子事务，或提供失败自动回滚的安全网，保证最终协作拓扑的完整性。 |
| **共享凭据导致多租户 Token 刷新竞态** | 多个 claude 实例共享同一份 OAuth credentials 软链接。一旦某个实例刷新 token 导致全局文件更新轮换，其余实例的内存 token 立刻失效从而报错 `Not logged in` 强制下线，且进程处于卡死态时不会自动重新读取文件。 | 1. 支持直接将全局刷新好的 token 作为环境变量（如 `CLAUDE_CODE_OAUTH_TOKEN`）直接在 spawn 时透传，避开文件；<br>2. 对处于 `Not logged in` 状态的 pane 触发 nudge 消息能重读文件，或在 daemon 引入全局 token 监听与内存级热更新同步。 |
| **本地与发布版本双源漂移与 doctor 盲区** | 二进制安装目录（installer）与全局 `cargo install` 容易发生不一致，且 systemd unit 中的 ExecStart 指向绝对路径导致升级后仍可能运行旧版 daemon。若 CLI 工具与 `ahd` 版本不匹配，其配置指纹漂移会误导 `ah up` 产生 realignment 风暴。 | 在 `ah doctor` 诊断输出中，显式列出「PATH 上生效的 ah/ahd 绝对路径与版本、systemd unit 实际指向的 ExecStart 二进制路径与版本」，并在发现双源漂移不一致时给予强告警提示。 |

## 第 4 节 · 角色分工事实归纳

### 4.1 各角色的事实职责

| 角色 | 实际承担职责 | 行为边界/特征 |
|---|---|---|
| **Master (主控 Claude)** | 1. 负责协作流程的核心调度，主动监控各 agent 的状态（如每分钟 capture-pane）；<br>2. 整合上下文并为子任务编写清晰的 brief，分发至 a1/a2/a3/a4；<br>3. 跑通完整的自驱闭环（SOP-08），并在集成阶段进行目标 level 的 E2E 物理验证；<br>4. 执行 mid-stream PR 的合并决策。 | **不写代码，不独自做领域分析。** 唯一的信号真理源是物理实证与捕获的实际 pane 内容，对任务挂起或漂移负总责。 |
| **PM (User / 开发者本人)** | 1. 定义大方向、需求挖掘与项目立项目标的敲定；<br>2. 对 milestone 级别的 squash merge 进行审核确认；<br>3. 接受并裁决主控上报（Escalate）的战略性变更（如契约崩溃、伪需求或调研大方向需要调整）。 | **不插手具体的工程细节、中间实现选型或日常合并决策。** |
| **Operator 代理 (PM 代理)** | 1. 独立于主控与日常开发，直接对 worker 派发独立的只读调研、审计或诊断类任务；<br>2. 进行任务的 Deconstruction（任务解构）并提供高层协调。 | **独立运行，不干扰 master 控制的 runtime 工作流。** |
| **a1 (Codex / 严谨操作者)** | 1. 专注于执行“严谨操作”任务：编写代码、调试编译报错、运行/回归测试、编写 TDD 失败用例；<br>2. 严格执行精确定位（如 `file:line` 引用），收集事实证据；<br>3. 独立审计其它 agent 产生的代码并提供 verdict 结论。 | **极强的机械精确度，弱于抽象推理。** 绝不让其处理过度发散的设计大方向问题。 |
| **a2 (Gemini / 分析与设计者)** | 1. 专注于执行“分析与设计”任务：概念调研、方案设计、系统建模、推理长日志；<br>2. 针对疑难领域问题和技术分歧，提供独立的评估报告与反方见解。 | **强于抽象推理和跨概念关联，极弱于机械精确。** 绝对不能在其 prompt 中写入“必须精确 file:line 引用”或硬性覆盖多章节的指令，防止其“印象派”幻觉。 |
| **a3 (Antigravity / 专注执行者)** | 1. 担任只读调研助手或专注的代码编写者；<br>2. 具备严格的 Worker 角色身份锁定（CLAUDE.md 中通过 `CCB_CALLER_ACTOR` 环境变量绑定识别）。 | **不充当 PM，不自主派单，完成指定任务即 reply。** |
| **a4 (Claude / 复合任务执行者)** | 1. 执行需要复杂语义理解、深度规则审计、集成测试场景的复合任务。 | **强语义理解与集成能力。** |

### 4.2 角色分工的协作铁律

1. **Worker 不自派单 (Workers Never Self-Dispatch)**
   Worker agent (a1/a2/a3/a4) 必须严格待在被派发的单次任务上下文中，禁止越权在沙箱内调用 `ah ask` 派活给其他 agent、禁止越权扮演 PM 或自主创建/推进 PR 周期。
2. **复合任务必须拆分 (Composite Tasks Must be Split)**
   严禁将包含“收集精确实证”和“基于事实抽象分析”的复合任务交由单个 agent 做全。必须遵循：**Codex 整理收集证据（强严谨） -> Gemini 针对该确凿证据进行推理评估（强分析）**。
3. **小机械 Design 派 Codex (Small Mechanical Design to Codex)**
   涉及局部 API 契约的微调、特定参数的值变更等小规模设计应直接指派给 a1 (Codex) 实施，从而规避 a2 (Gemini) 产生过度设计和范围漂移。
4. **上下文完全重置与独立 (Context Isolation)**
   每个新任务派发前，必须通过 `/new` 或 `autonew` 重置该 agent 的 session，彻底避免历史 session 内存污染对新任务产生未知干扰。
5. **角色边界每次派单硬提醒 (Hard Role Backstop)**
   因 LLM 在缺少 onboarding 或空白的沙箱中容易漂移，除了在项目 `CLAUDE.md` 中以条件式定义身份外，主控每次 dispatch worker 时必须在 brief 头部显式注入 worker 角色约束与“禁止自主派单、禁止做 PM 决策”的安全护栏。
6. **Master-PM 决策红线**
   主控在自驱闭环（SOP-08）中拥有对实现路径、PR 合并、测试手段的完全自决权。**严禁问用户工程细节**、实现选型或 "PR 是否要合" 等过程问题。只有立项契约崩溃、目标需大范围更改或多次尝试失败时才能触发 Escalate 升级给 PM。
7. **双重物理验证 (Physical Evidence & Recapture)**
   不信赖任何 agent 产出的 Self-report（如 INVENTORY.md）。主控必须对产出执行物理 ls/wc 验证；在等待 worker 时，主控必须每分钟亲自 capture-pane 校验实际屏幕，不能依赖任何二次事件信号。
