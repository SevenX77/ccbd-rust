# 编程场景配置包工程落地理论 (a3-engineering-theory-v2.md)

本文件由 a3 worker (explore/research 角色) 独立撰写。我们抛弃华丽的学术命名与“重度重试+测试判定”的理想化模型，直接站在**产品经理（PM）与主控智能体（Master）**的实际开发视角，探索如何将「没有测试能判的架构不确定性」转化为「便宜模型能照着做的确定性工程实体」。

---

## 理论核心：harness 的本质是“不确定性的结构化转译”

我们用信息论做参考，绝不是为了在运行时算出一个虚无的“熵值”，而是将它当作一个**指南针**。
* **问题所在**：便宜模型（如 DeepSeek-V3 等）不缺代码语法能力，缺的是“大局观”与“架构直觉”。它不知道什么叫“优雅的解耦”，也不知道怎么防止“a 出问题改 a，结果跟共享底座 c 冲突”。
* **解决之道 (harness 的职责)**：harness 不去教模型怎么思考架构，而是通过**强制性的结构与流程**，把人类脑中“模糊的设计想法”和“看不见的全局耦合”，在写代码之前，转译成**便宜模型只需“查表/核对”就能 100% 正确执行的确定性活**。

以下是我们将这套原理落地的 5 个具体工程组件设计：

---

## 一、 架构门禁：用“设计答卷”代替“编译器”

* **痛点**：代码该不该这样抽象、接口有没有充分解耦、SWR 缓存 Key 划得对不对……这些架构决策**没有编译器和测试能卡住**，便宜模型写出烂代码一样能跑通测试。
* **harness 落地实体 —— 「设计契约答卷 (Design Contract Questionnaire)」**：
  在 Worker 动手写代码前，harness 强制要求 Worker 填写一份结构化的 JSON 答卷（如 `design_intent.json`），回答以下架构关键点：
  1. 本次新增/修改的 API 路由是什么？
  2. 使用的 SWR 缓存 Key 是什么？（强制格式核对）
  3. 修改了哪个共享底座文件？是否引入了新的全局 import？
* **机器校验法**：
  harness 运行一个极其简单的**静态契约校验器（Schema Linter）**来批改这份答卷，而不是直接批改代码。例如：
  * 校验器读取 `ah.toml` 中的白名单，检查答卷中是否有“在 `components/` 下直接 import `tauri` API”的声明；
  * 检查 SWR 缓存 Key 是否符合“层级命名规范”。
* **效果**：将“不可测的代码架构直觉”转化为“可被 Linter 硬卡死的 JSON 答卷比对”。不提交合规答卷，Worker 就无法获得写代码的授权。

---

## 二、 前端对齐：用“结构化质疑器”逼出设计细节

* **痛点**：非技术背景的 PM（用户）提需求往往只有一句话（“我想要个七牛卡片”），便宜模型拿到需求就直接盲目堆砌 UI 代码，漏掉各种交互边界（如 loading 态、报错态、空数据态），导致后期疯狂打补丁。
* **harness 落地实体 —— 「交互决策挑战器 (UI Boundary Interrogator)」**：
  在 Master 接收需求并划分工单时，harness 自动拦截，并强行弹出一份**「未决决策清单」**模板给用户。清单强行要求用户/Master 明确以下 4 个前端物理事实：
  1. **异常态**：当接口网络报错或鉴权失效时，UI 展示什么？（提示：弹 toast 还是显示 inline 错误？）
  2. **加载态**：数据在拉取过程中，UI 显示什么骨架屏？
  3. **空态**：当用户还没有配过七牛 key 时，界面显示什么？
  4. **组件复用**：当前已有哪些 Button/Select/Card 可以直接套用？
* **效果**：在动手前，强行用结构化提问把用户脑中的模糊想法“挤压”成确定的文案与路由。便宜模型只需照着这份确定的清单“还原”UI 即可，不再有瞎猜的空间。

---

## 三、 耦合显性化：用“联动依赖地图”消除暗雷

* **痛点**：改了 A 文件的接口，共用底座 C 的 B 模块跟着坏了，这种联动在写代码前便宜模型根本看不见。
* **harness 落地实体 —— 「影响范围追踪器 (Impact Range Tracker)」**：
  在 Worker 领到工单（如“修改 `llm.py` 中 credentials 接口”）的一瞬间，harness 自动运行一个静态 import 解析脚本，生成一份**「依赖影响报告」**，直接焊死在 Worker 的 Prompt 头部：
  > “⚠️ 警告：你正在修改的文件被以下组件引用：`LlmRolesTab.tsx` (前端卡片)、`test_llm_registry_api.py` (后端测试)。你必须保证修改后的 JSON Schema 能够完美兼容上述文件，否则修改将被门禁直接驳回。”
* **效果**：把原本隐藏在代码深处的复杂耦合，转化为**一张拍在便宜模型眼前的“只准修改本文件，必须兼容这些关联文件”的硬性清单**。

---

## 四、 范围切分：单向单输出的“极简切片法则”

* **痛点**：怎么确定一个任务已经小到了便宜模型能够轻松拿捏的程度？
* **harness 落地实体 —— 「单输出流水线 (Single-Output Pipeline)」**：
  Master 拆分工单时，harness 强制执行一条**物理切割红线**：
  > **「任何给到 Worker 的子任务，其预期的物理输出（改动文件）必须具有单一的职责类型，且原则上每次只改动 1 个文件，最多不超过 3 个同模块文件。」**
  * **反例**：“请修改后端 `llm.py` 接口，同时修改前端组件展示它。” —— 这是一个双职责任务（跨越了前后端两层信道），便宜模型必乱。
  * **正例**：拆分为两个子任务：
    1. 【后端子任务】修改 `llm.py`，暴露新字段。输出：仅 `llm.py`。
    2. 【前端子任务】修改组件，消费新字段。输出：仅对应的 `.tsx`。
* **效果**：通过限制“物理输出文件的个数与类型”，强行把任务范围压缩在便宜模型的上下文舒适区内。

---

## 五、 自我沉淀：让“踩坑历史”转化为“门禁规则”的自愈飞轮

* **痛点**：写在 `CLAUDE.md` 或 memory 里的规矩（比如“串行跑测试防 OOM”）很脆，AI 开发时间一长就容易忘掉，规则流于形式。
* **harness 落地实体 —— 「负反馈硬化机制 (Negative Feedback Hardening)」**：
  我们不靠规则来约束，我们靠**“把痛点写成脚本和门禁配置”**来解决。
  * *实操*：我们在 ccbd-rust 中踩过“并发跑 cargo 会 OOM 崩掉 ahd”的坑。
    * *软规矩（易忘）*：在 memory 里写“请大家以后串行跑测试”。
    * *harness 硬化（自沉淀）*：在 `ah.toml` 或 Makefile 中直接锁死 `alias cargo-test='cargo test -- --test-threads=1'`，并在 CI 里加上 `CARGO_BUILD_JOBS=1` 环境变量。
  * *实操*：AI 总是漏掉 `CLAUDE_CONFIG_DIR` 必须是绝对路径这一条。
    * *harness 硬化*：我们在被测 daemon 启动入口写一段 5 行的 Shell 脚本检测，如果是相对路径直接退出并报“FatalError”。
* **效果**：飞轮自我改进的真身，就是**“每一次踩坑，就往编译/CI/脚本里增加一行硬门禁代码”**。智能体和人类在未来的任务中自然而然地被这些物理护栏推着走，不需要去记忆任何规则。

---

## 六、 落地建议表：给 SCS 每一站装上具体可造的 harness 实体

如果要实现用便宜模型替代 Claude，我们应该在 SCS 的级联信道中这样布置我们的 harness 硬件：

1. **第一站 (意图→Spec)**：
   * **harness 硬件**：**「交互决策挑战器」**。用户提模糊想法时，自动拦截并弹出 4 问表单，逼出异常态、加载态设计，写成 markdown spec。
2. **第二站 (Spec→Code)**：
   * **harness 硬件**：**「设计契约答卷」**与**「影响范围追踪器」**。动手前给 Worker 贴上关联组件清单，动手时逼 Worker 填 schema 答卷，本地 pre-commit 跑 Schema Linter 批改。
3. **第三站 (Code→物理运行)**：
   * **harness 硬件**：**「单输出流水线」**与**「轻量变异门禁」**。任务强制切到单文件修改级别，跑本地 ruff/eslint 物理校验。
4. **反馈路由阶段**：
   * **harness 硬件**：**「二分法硬路由」**。测试失败时，若 Spec 目录下存在未决决策或答卷校验失败，直接回弹到 Spec 重修阶段；若答卷全绿，只允许 Worker 在当前 worktree 改单文件代码。
5. **自我沉淀横切**：
   * **harness 硬件**：**「门禁自硬化脚本」**。每次 PR 合并前，检测是否有新增的常规范性问题，自动更新项目的 `.eslintrc` 或 `import-linter` 配置文件。
