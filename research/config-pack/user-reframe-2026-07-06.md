# 用户原话(2026-07-06,配置包方向对齐,逐字留底)

> 本文件是用户亲口对"编程场景配置包"目标的澄清与更正,给 a3 做设计的第一手真相源。不要转述、不要脑补,以此为准。

## 用户消息一(定目标)

他应该就是一个专业开发的 agent harness,他的终极形态是 spec coding system(coding 里面另一个库,做过一些研究,讨论过一些框架,但还没有开始真正落地,ah 原本就是为他打造的一个基础设施)。但是现在我没有想做的那么深,现在的目标是在 **1 master + several workers 这样一个体量里面,把单个编程任务的工程质量尽可能做到最好**。

最近我一直在开发 agent-harness 这个项目,我赶着把他的前端打通让他尽快能够上线。一开始我一直用我们这套 ah 的 sop 开发:写设计 spec、交叉审核、写 tdd 测试、实施、审计验证。我坚信 spec coding 理念,甚至花一周整出一整套 MVP1 alignment。但全部实施完进入实际 app 测试、前端优化时,发现**实施质量离我想象差距巨大**,很多地方根本没按 MVP1 设计目标落实,很多细节实施时也没考虑到。现在这阶段我用的是一个快速 sop 流程(见 agent-harness 最新 handoff,也是一套 sop 但给单个 agent 用)。随项目复杂度提升,这方式效率越来越低、越来越乱、按下葫芦起了瓢(刚更新 T1-T4 也和一个功能一直修不好越修越差有关)。

这种情况下我感受最深:**fable5 确实吊打 opus4.8 和 codex gpt5.5**。fable5 理解问题设计问题更精准、更贴合原意、听得懂人话、不乱改已正确的代码、一击命中。其他模型反复改,时间和 token 是 fable5 的 5 倍。但 fable5 太贵。我想试试 **ah 开发流程的准确率、问题解决率,是否能勾到 fable5 的水准**。

开发中感受最深的是:**软件开发的架构层非常重要**(如前端获取后端数据的策略);**模块规划解耦复用也非常重要**(如原子测试 atom probe 解决了很多测试问题)。**AI 总是用打补丁的方式,a 出问题改 a,不会深究 a、b 同时使用的 c 有问题**,结果改了 a、b 还是有问题,还留下一套和 c 可能冲突的功能。

这也是产品适用场景:**我(用户)不是资深工程师**,根本不懂也无法提前规划系统框架、模块规划。我只能站产品经理视角:碰到问题知道产品行为应该怎样,甚至能理解后端底层运行原理该怎样(这点已比多数用户好)。现在 vibe coding 工具也是走到哪做到哪,没有全局思维。**我希望 agent 有全局思维,帮用户补齐资深工程师的架构能力**——这目标可能有点大,是 scs 的终极目标。当前目标:通过这套 agent harness + sop **达到 fable5 的质量**。突然想到:**配置里要加上模型**。

## 用户消息二(更正我的判断,六条)

1. **spec 很好**,基本到现在还是黄金铁律,没什么问题。出问题的地方是**我没看到真实的东西想象不到细节问题,设计得不够细致**。
2. 当初我其实**没走最严格的重型流程**,没全用 ccb(当时 ccb 设计太重了:三个 agent 吵架收敛、每步都审核,太慢)。我走的是:根据设计文档分几大模块,一个个大模块让 agent 一批批实现,agent 也拆成几批任务,走 kiro spec、tdd,最后也交叉验证审核。**当时最大的问题是:因为大多任务以后端为重、附带实现前端,不是按前端用户线索一个个功能去实现,所以没有严格 e2e 测试,我也没有直接去验收每一个功能、前端去点、亲眼看**。然后**设计层面缺失了我说的架构层的工程设计,我的经验盲区**。这不是用 fable5 能解决的问题,这是**根本没有人提到要从顶层框架上补齐"因我架构经验缺失而导致的工程架构问题"**。
3. 前后是两个问题,我不是说 fable5 能解决前面的问题。**fable5 的对比场景是:我已经在一个功能一个功能啃的阶段、用单 agent 快速流程时,fable5 远远好于其他流程**。我现在回头整理这套多 agent sop,是因为我觉得**这套流程的结果在单个功能实现上不比 fable5 差**。比如 ah 的开发我也基本托管状态,但一个个功能实现得比较扎实。也可能因为复杂度还没到 agent-harness 这个级别。
4. 希望你(主控)不要自己闷头忙活,**要用起 antigravity,两个 agent 互相启发或找茬,永远大于一个 agent 闷头钻死胡同**。
5. **谁来帮我填补资深架构工程师经验不足的问题?谁来盯着每一个功能实现的事件订阅、数据规范、API 规范、有没有充分解耦等问题?这可能是 spec coding system 的设计大问题,并不是 master+workers 工作小组能解决的**。我们可以**先讨论大问题,然后把工作小组在这个大框架里的位置摆对,设计不要偏离大框架**。
6. 现在 master+workers 的配置还有很多可优化的地方:**哪些可以沉淀成 skill,哪些需要用 hooks 保证可靠,哪些能力需要 mcp server,特定项目的工程规范怎么守住**。现在只靠 claude.md 和 memory,**很脆,容易失焦**。
