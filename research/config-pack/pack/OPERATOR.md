# OPERATOR.md — operator 的任务推进保障体系

operator 的角色定位、职责与判断框架见 `OPERATOR-HANDBOOK.md`「角色与判断(operator 内核)」章;本文件是其中「监督」职责的哨兵机制实现。

GUIDE §四是逐条实践;本文件回答一个问题:**怎么保证派下去的任务能顺利跑到底**。
核心结论(多代实测):靠自律和"记得回来看"必然停摆,靠机制才能闭环。

## 一、第一性原则:每个等待都必须有闹钟

任何角色(master 或 operator)进入"等某事完成"状态的那一刻,必须先挂上一个
**会物理唤醒自己的机制**,才允许把注意力移开。没有闹钟的等待 = 停摆事故预定。

实测失败史:规则文档写"必须持续监控"→ master 照样裸等 3 次(说"5 分钟后回来看"
然后永远不回来);规则是纪律,纪律对 LLM agent 不可靠,**机制才可靠**。

## 二、三层哨兵体系(按代价从低到高互补)

1. **每单哨兵(master 层,强制)**:派单拿到 job_id 后,立刻挂后台任务
   `timeout <预算> <编排器的阻塞等待原语> <job_id>`(ah 是 `ah pend`),预算 =
   时长估计 ×2(下限 15min)。后台任务退出自动唤醒:正常退出 = job 收口(去产物轨
   亲验,不信 job 状态);超时退出 = 停摆警报(先看 pane 真相再分诊)。
   **没挂哨兵不许结束回合;同时在途 N 单就挂 N 个。**
2. **全局停摆哨兵(operator 层,常驻)**:一个 90s 周期的体检脚本,只在异常态告警:
   (a) 有 QUEUED/DISPATCHED job 且全部 agent 空闲 = 有活没人干;
   (b) 同一 job 停 QUEUED ≥5min = 派发被卡;
   (c) master pane 连续 3 拍零变化、有未收口 job **且无任何 agent BUSY** = master 裸等
   (有 agent 在干活时 master 安静是正常态,漏这个前置条件会持续误报,实测);
   (d) **无在途 job + 产物轨(git HEAD/约定落盘)整窗不动 + master pane 冻结 = 调度停摆**
   ——这个形态长得和「干完了」一模一样,必须显式设防(实测:master 一棒收尾后静默
   6 小时,所有"有活没人干"类规则都不覆盖它)。
   状态变化才发声(去重),恢复也报一行。这是兜 master 哨兵失效的外层网。
3. **状态翻转监听(operator 层,常驻)**:轮询编排器 DB 的 job/agent 状态做 diff,
   任何翻转(创建/COMPLETED/CRASHED/异常态)推一条事件。这是"发生了什么"的流水账,
   与第 2 层的"该发生的没发生"互补。

**哨兵必须用心跳模型,不用「挂一次长跑到底」**:长跑型哨兵有两个致命属性——
①它无法自证存活(挂了/被宿主杀了/规则有盲区,外界看都是同一种安静);②它只在
「自认为异常」时发声,规则盲区=永久沉默。心跳模型(固定节拍主动退出报状态,正常
也退,退出即唤醒 operator,收到即续挂)让「哨兵还活着吗、它看到了什么」每个节拍
都有答案。实测教训:一只长跑哨兵因规则盲区(上面的 (d) 形态)静默 6 小时,operator
误以为有哨兵在岗——盲区 + 无法自证存活叠加,和 master 停摆同时发生,双层失守。

## 三、停摆分诊树(哨兵响了之后)

先取物理真相(capture-pane 看 agent 实际内容,git log/status 看产物),再分诊:

- **活干完了 job 没翻**(语义假完成,部分 provider 的 harness 后台化所致):
  状态作废,产物为准。产物已收口 → cancel 该僵尸单释放通道;产物没收口 → 不动,
  agent 上下文完好,等真产出,**绝不重派**(重派 = 双工浪费 + 队列堆积)。
- **占道**:同 agent 有陈旧在途单,新单被静默排队(实测:daemon 可能零日志)。
  处置同上——先清占道的僵尸单。
- **cancel 卡 CANCEL_REQUESTED 不落地**(无认领人):升级为 kill 该 agent + up 重建。
  注意 up 的连带(见下),重派 brief 必须自包含。
- **prompt 根本没落 pane**(派单竞态/幽灵文本卡扫描器):job 有、pane 空。
  cancel → kill → up → 等 IDLE → 自包含重派。从未开始过,无部分完成之忧。
- **agent 真死/真卡**:走对应恢复 SOP;死因当日记观察日志。

## 四、高危操作的已知连带(动手前过一遍)

- **改过 ah.toml 再跑 up = 全员 REALIGN**:所有 agent 上下文清零重建;实测还出过
  非原子丢 agent(4/5 拓扑悄缩)与 respawn 会话错命名。跑 up 前先核对配置漂移面;
  跑完必双验拓扑(编排器视角 + tmux list-panes 视角对照);**定位 agent 一律用
  pane id,永不信 session 名**。
- **ah.toml 的运行时注入是未提交状态,git 清理会把它无声抹掉**:operator 注入的
  机器本地 env(代理、工具链 HOME、跳权限 cmd)按纪律不入库,于是它在 worktree 里
  就是一段永久 dirty diff——任何 agent 顺手 `git checkout .`/`git restore` 做"工作区
  清理"都会把注入清零,且症状延迟发作:token/env 物化只在 `ah start` 时发生,当场
  看不出异常,直到下一次 respawn/up 席位才批量秒死(实测 2026-07-10:一次清理导致
  全部 agy 席无代理可用,只能重注入 + 整栈重启)。防线:worker 规则里明写**绝不对
  ah.toml 做 checkout/restore,commit 一律用显式文件列表**;operator 在栈根留注入
  内容的副本,重启前先 diff 核对注入还在。
- **无分支保护的仓库,auto-merge = 秒合不等 CI**:绝不用 `--auto`;挂 CI 看门
  (逐 check 事件 + 全绿/有挂终态),全绿后普通 merge。CI(并行测试 + 跨平台)是
  唯一全量门,本地串行绿不算数。
- **机器级 cargo 槽只有一个**(内存约束):派任何含编译/测试的单之前,全机 ps
  核查没有别的 cargo 在跑;两条泳道的收口窗口要错开排队。
- **换血(换编排器二进制)必须成对换**(CLI + daemon 从同一构建),每个大模块合入
  后换一次,换完的下一段 dogfood 就是疗效观察窗,病种发生率按代次记账。执行换血时
  知会用户(预授权也要报边界),换血前对活栈务必同源二进制(CLI/daemon 同一构建),
  换完双验拓扑。
- **每代必出阶段性疗效判决报告**(建议 `logs/gen-efficacy-reports.md`):一个观察
  窗关闭时,逐条对照"预期效果 → 实际观测 → verdict",verdict 用五级(治愈/改善/
  无变化/恶化/未观测)。**未观测不许算治愈**——没在活栈见到病种发生率下降,就只能记
  "未观测",不能凭 merge/CI 绿宣称修好。关窗即把判决推送用户,不等被问。这本账是
  "merge≠完成、实证闭环才算"的落地凭证,也是决定某修法要不要回炉的依据。

- **composer 里的文本永远不是"裁决存在"的证据,代投前必须过可审计通道核实**:
  「打字不提交」病会在 worker composer 里留下酷似 master 裁决的未提交文本,operator
  按病处置(C-u+重注入)时若给它加上「代投 master 裁决」标注,就是在**凭空制造授权**
  ——实测 2026-07-11:operator 把一条 master 从未作出的"Q1 授权"当丢失投递代投给
  gatekeeper,gatekeeper 无从核实来源只能采信,据此提交了(碰巧客观正确的)修法;
  master 复盘时发现裁决无出处,全链路停摆等 operator 对质。纪律:代投任何"决定/授权"
  之前,先在可审计通道(编排器 job/reply 记录、落盘裁决文件)里找到它真实存在的证据;
  找不到就把它当**未决问题**退回 master 重新裁决,绝不补授权标注。哨兵自动 Enter 只
  许用于疏通投递,永不用于代授权。事后补救姿势:不回滚客观正确的技术改动,由 master
  对该改动**事后正式裁决**(追认或否决)并把来源核实经过写进消解报告,程序瑕疵用
  程序补正。

## 五、闭环的证据纪律

- **job 状态/字段一律只当提示**:状态会撒谎(假 COMPLETED / 永久 DISPATCHED),
  reply 载荷也会错(实测存成 brief 残片)。可信证据只有三样:pane 实际内容、
  git 产物、约定落盘文件。
- **agent 说"完成"必须物理验证**:ls/wc/git log 落盘核验,自报清单会造假。
- **任何异常当日入观察日志**(四要素:症状/根因/处置/状态),这本账是后续
  结构决策(删推断路径、改协议)的唯一弹药,也是换血疗效对照的基线。

## 六、operator 自身的位置

operator 只动脑不动手(GUIDE §四.7),但**哨兵体系是 operator 亲手维护的例外**——
它是控制回路本身,不是业务执行。每次接手一个栈,第一件事就是把三层哨兵架起来;
每次发现新停摆模式,先问"哪一层哨兵该抓住它而没抓住",修哨兵优先于修个案。
