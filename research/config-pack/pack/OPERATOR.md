# OPERATOR.md — operator 的任务推进保障体系

GUIDE §四是逐条实践;本文件回答一个问题:**怎么保证派下去的任务能顺利跑到底**。
核心结论(多代实测):靠自律和"记得回来看"必然停摆,靠机制才能闭环。

## 一、第一性原则:每个等待都必须有闹钟

任何角色(master 或 operator)进入"等某事完成"状态的那一刻,必须先挂上一个
**会物理唤醒自己的机制**,才允许把注意力移开。没有闹钟的等待 = 停摆事故预定。

实测失败史:规则文档写"必须持续监控"→ master 照样裸等 3 次(说"5 分钟后回来看"
然后永远不回来);规则是纪律,纪律对 LLM agent 不可靠,**机制才可靠**。

## 二、三层哨兵体系(按代价从低到高互补)

1. **每单哨兵(master 层,强制)**:派单拿到 job_id 后,立刻挂后台任务
   `timeout <预算> <编排器的阻塞等待原语> <job_id>`(ah 是 `ah pend`),预算 =
   时长估计 ×2(下限 15min)。后台任务退出自动唤醒:正常退出 = job 收口(去产物轨
   亲验,不信 job 状态);超时退出 = 停摆警报(先看 pane 真相再分诊)。
   **没挂哨兵不许结束回合;同时在途 N 单就挂 N 个。**
2. **全局停摆哨兵(operator 层,常驻)**:一个 90s 周期的体检脚本,只在异常态告警:
   (a) 有 QUEUED/DISPATCHED job 且全部 agent 空闲 = 有活没人干;
   (b) 同一 job 停 QUEUED ≥5min = 派发被卡;
   (c) master pane 连续 3 拍零变化且有未收口 job = master 裸等。
   状态变化才发声(去重),恢复也报一行。这是兜 master 哨兵失效的外层网。
3. **状态翻转监听(operator 层,常驻)**:轮询编排器 DB 的 job/agent 状态做 diff,
   任何翻转(创建/COMPLETED/CRASHED/异常态)推一条事件。这是"发生了什么"的流水账,
   与第 2 层的"该发生的没发生"互补。

## 三、停摆分诊树(哨兵响了之后)

先取物理真相(capture-pane 看 agent 实际内容,git log/status 看产物),再分诊:

- **活干完了 job 没翻**(语义假完成,部分 provider 的 harness 后台化所致):
  状态作废,产物为准。产物已收口 → cancel 该僵尸单释放通道;产物没收口 → 不动,
  agent 上下文完好,等真产出,**绝不重派**(重派 = 双工浪费 + 队列堆积)。
- **占道**:同 agent 有陈旧在途单,新单被静默排队(实测:daemon 可能零日志)。
  处置同上——先清占道的僵尸单。
- **cancel 卡 CANCEL_REQUESTED 不落地**(无认领人):升级为 kill 该 agent + up 重建。
  注意 up 的连带(见下),重派 brief 必须自包含。
- **prompt 根本没落 pane**(派单竞态/幽灵文本卡扫描器):job 有、pane 空。
  cancel → kill → up → 等 IDLE → 自包含重派。从未开始过,无部分完成之忧。
- **agent 真死/真卡**:走对应恢复 SOP;死因当日记观察日志。

## 四、高危操作的已知连带(动手前过一遍)

- **改过 ah.toml 再跑 up = 全员 REALIGN**:所有 agent 上下文清零重建;实测还出过
  非原子丢 agent(4/5 拓扑悄缩)与 respawn 会话错命名。跑 up 前先核对配置漂移面;
  跑完必双验拓扑(编排器视角 + tmux list-panes 视角对照);**定位 agent 一律用
  pane id,永不信 session 名**。
- **无分支保护的仓库,auto-merge = 秒合不等 CI**:绝不用 `--auto`;挂 CI 看门
  (逐 check 事件 + 全绿/有挂终态),全绿后普通 merge。CI(并行测试 + 跨平台)是
  唯一全量门,本地串行绿不算数。
- **机器级 cargo 槽只有一个**(内存约束):派任何含编译/测试的单之前,全机 ps
  核查没有别的 cargo 在跑;两条泳道的收口窗口要错开排队。
- **换血(换编排器二进制)必须成对换**(CLI + daemon 从同一构建),每个大模块合入
  后换一次,换完的下一段 dogfood 就是疗效观察窗,病种发生率按代次记账。

## 五、闭环的证据纪律

- **job 状态/字段一律只当提示**:状态会撒谎(假 COMPLETED / 永久 DISPATCHED),
  reply 载荷也会错(实测存成 brief 残片)。可信证据只有三样:pane 实际内容、
  git 产物、约定落盘文件。
- **agent 说"完成"必须物理验证**:ls/wc/git log 落盘核验,自报清单会造假。
- **任何异常当日入观察日志**(四要素:症状/根因/处置/状态),这本账是后续
  结构决策(删推断路径、改协议)的唯一弹药,也是换血疗效对照的基线。

## 六、operator 自身的位置

operator 只动脑不动手(GUIDE §四.7),但**哨兵体系是 operator 亲手维护的例外**——
它是控制回路本身,不是业务执行。每次接手一个栈,第一件事就是把三层哨兵架起来;
每次发现新停摆模式,先问"哪一层哨兵该抓住它而没抓住",修哨兵优先于修个案。
