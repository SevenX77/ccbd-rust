# VERIFY.md · 项目验证档案(fill-once)

> **这是什么**:本项目"构建 / 测试 / 验收"的唯一权威档案。接入配置包时填一次,之后所有 agent(master 派单 brief、worker 执行、审阅者复核)只引用这里,**不允许现场重新推导编译测试命令**。
>
> **为什么存在**:没有沉淀时,每个 agent 每次任务都要重想一遍"怎么编译、跑哪个测试、带什么参数"——推导会出错(跑错测试包、漏资源约束参数、白烧编译),且错误每次任务重复发生。档案错了改档案,一处生效。

---

## 1. 构建与测试命令(⚠ 全部替换成你项目的真实命令)

| 用途 | 命令 | 何时用 |
|---|---|---|
| 构建 | `⚠ 例: CARGO_BUILD_JOBS=1 cargo build` | 需要产物时 |
| **迭代测试(定向)** | `⚠ 例: CCB_TEST_SKIP_REAL_PROVIDER=1 CARGO_BUILD_JOBS=1 cargo test --test <受影响测试文件> -- --test-threads=1` | **红绿循环中间的每一轮**。只跑受本次改动影响的测试文件/模块 |
| **收口测试(全量)** | `⚠ 例: CCB_TEST_SKIP_REAL_PROVIDER=1 CARGO_BUILD_JOBS=1 cargo test -- --test-threads=1` | **任务收尾跑一次**,作为交付门;严禁用定向结果替代 |
| 编译级全量校验 | `⚠ 例: cargo test --no-run` | 删/改公共符号后,确认所有引用(含集成测试)能编译 |

**迭代/收口分离纪律**:红绿循环中间只跑定向命令;全量只在收口跑一次。中间反复跑全量 = 每轮多烧数分钟编译;迭代期用"按测试名过滤"容易跑错测试包白烧一轮,**定向必须用"按文件/模块"的形式**(如 `--test <file>`),按名过滤仅在确认测试所在包时使用。

## 2. 资源与环境约束(⚠ 替换)

- `⚠ 例: 7.7G 内存 VPS,并行构建会 OOM——所有构建/测试必须串行(CARGO_BUILD_JOBS=1 + --test-threads=1),同一时刻全机只允许一个构建进程`
- `⚠ 例: 真实 provider 网络测试默认跳过(CCB_TEST_SKIP_REAL_PROVIDER=1),CI 才跑`
- `⚠ 例: 本地串行绿 ≠ 并发安全;CI 默认并行,CI 绿才是并发正确性的真验收`

## 3. 任务完成后的验收矩阵(按改动类型查表)

| 改动类型 | 验收方式 | 工具 | 人参与? |
|---|---|---|---|
| 纯逻辑 / 后端 / 库代码 | 自动化:单测 + 集成 + 收口全量 | 项目测试框架 | **否,全自动** |
| CLI / daemon 行为 | 自动化 e2e(隔离环境,绝不在活栈/生产实例上测)| `⚠ 例: 独立 STATE_DIR + 独立 socket + trap 清理` | 否 |
| **前端 / UI** | **按验收开关二选一**(派单 brief 里必须显式指定):<br>`agent` 模式 = agent 自验 | `⚠ 例: playwright(DOM 断言)/ computer-use(视觉验收)` | 否 |
| | `user` 模式 = 用户亲验(用户在场时更快更准) | 起本地服务 + 给出访问入口,停下等用户签收 | **是** |
| 文档 / 纯 markdown | 物理实证:落盘路径 + wc/grep 关键内容 | ls / wc / grep | 否 |

**验收开关(acceptance-mode)**:涉及 UI 的任务,master 派单时必须在 brief 里写明本单是 `agent` 还是 `user` 验收,默认 `⚠ 你项目的默认值`;用户不在场时不允许选 `user` 然后干等。

## 4. 红灯处置(收口全量红了怎么办)

1. 先证伪:对照主干单跑该测试,分清"既有红"还是"本次引入"。
2. 既有红 → 记录证据(主干同红的输出),不阻塞交付,单独立项修。
3. 本次引入 → 回红绿循环修,不允许"看起来无关"就跳过。
4. flake 判定:`⚠ 例: 同 commit 并行任务一红一绿 = flake 签名,重跑一次并留证`。
