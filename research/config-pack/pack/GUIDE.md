# 编程场景协作指南

这份指南是配置包的「人读层」——讲清楚用 ah 编排 agent 做工程时,整套协作怎么跑。规则文档(`.ah/rules/*.md`)是注入给 agent 的;这份指南是给你(搭这套系统的人)和你的 operator 代理看的。

---

## 一、三层协作拓扑

```
用户(需求/目标层)
   ↕  只在需求与方向上参与
operator(人的代理)
   ↕  转达目标、盯结果、管 CI/发布
master / 协调者(ah 托管)
   ↕  ah ask 分派 / ah pend 收结果
workers:实现者 ×N · 设计者 · 审计者 …(数量/provider 你定)
```

- **用户**:定大方向、需求、立项目标;裁决升级上来的战略变更。不插手工程细节。
- **operator(人的代理)**:把用户目标翻译成 brief 转给 master;持续盯到真出可信结果;独占 CI/PR/发布权。是可选但强烈推荐的一层——它把用户从「盯实现」里解放出来。
- **master/协调者**:规划、错峰排期、分派、设计辩论、审阅、收敛。
- **workers**:只执行被指派的单条任务,完成即回、等下一单。

> 各角色的具体规矩在对应 `.ah/rules/<id>.md` 里。这里讲的是它们**怎么协同**。

---

## 二、SOP 协作闭环

一个目标进来,自驱跑完这个环,不闭合就回上一步:

```
调研(设计者/实现者 sweep 素材)
  → 设计(架构级走「设计管线」见下章;小机械设计直接派实现者)
  → 实施(实现者 TDD 红绿)
  → 审阅(交叉严审 + 审计者独立审;逼 baseline 对照;绝不同实例自审)
  → 集成/e2e(审计者端到端验证,绝不在活栈/生产实例上测)
  → 目标验证(master:达成了用户目标吗?)
  → 没闭合 → 回到调研/设计;闭合 → 收敛交付
```

铁律:
- **抓到 blocker 先修**,不包装成「放哪个 PR / 要不要修」抛给用户等拍板。commit 落点是工程细节,自决。
- **只有真的产品方向选择才升级**给用户。
- **阶段转换处停下作报告**(现状/根因/下一步),让用户决下一步方向——这不算「停下来问」。

## 三、设计管线(架构级课题专用)

架构级课题(新子系统、跨模块契约、恢复/安全语义等)不许一个 agent 包办「设计+spec+实施」。走五步管线,**每步换手是刻意的**——分工依据见 ROLES.md「设计权在谁手里」:

```
1. 设计者主笔设计(零方向 brief)
2. master 带事实论据辩论 → 收敛 → 冻结
3. 实现者把冻结设计转写成 spec + TDD 框线
4. 交叉严审 spec(另一个实现者实例)
5. 设计者对抗审 spec 忠实度(FAITHFUL 才放行)
```

每步的关键纪律:

1. **零方向 brief**:只陈述处境事实和目标,不给结论、不给答案形状、不指定「读什么按什么做」;约束只防偷懒(落盘路径/证据要求/覆盖面/severity 分级)。必带显式反讨好条款+推翻问法授权+markdown-only 护栏。operator 审每份设计 brief 后才发。
2. **辩论到收敛**:master 逐条核验设计的引用与断言(亲手 grep,抓引用不存在的列/行);**每个安全关键断言必须逼出机制或证据,「此处安全」的表态不收**;设计者有据让步/有据坚守都合格,讨好式全盘接受=失败。**安全关键残留必须压回设计本体才准冻结**,不许留给下游「实施时再说」——危险决定被下游软化是实测出过活栈事故的病。
3. **冻结稿是权威**:spec/实施与设计冲突时,以冻结稿为准;要改必须回到辩论层走修订,**spec 不许单方面稀释设计**(加例外、放宽 fail-closed、降低门槛都算稀释)。
4. **交叉严审**:另一个实现者实例专盯 spec 对设计的偏离与技术漏洞。
5. **忠实度对抗审**:设计者回头审 spec,对抗焦点=「被软化/被例外化」模式,给明确 verdict。
6. **双盲互评变体**(重大评估类课题,如全项目架构体检):主控与设计者各自独立评估→互相裁决对方报告→收敛为联合终稿。三条纪律:(a) 双盲 brief 只给问题与处境事实,**不泄任何一方的结论**(泄了=回音壁,发散值归零);(b) 收敛裁决时,对对方**独有的**关键断言必须亲手核验(grep/读码)再判 AGREE/REVISE,不许凭印象;(c) **允许「第三真相」**——双方可能各对一半,真答案在两者之外(实测:一方称「无熔断」、另一方称「有熔断」,亲验发现熔断存在但被成功路径清零击穿,双方都没看全)。

**错峰排期**:辩论花时间,所以它必须与实施窗口重叠——实现者在做上一单实施/测试时,master+设计者并行推进下一个设计课题。串行空等是 PM 失职。

## 四、operator ↔ master 代理实践

如果你用一个 operator 代理去驱动 master(强烈推荐),这些是实操要点:

1. **不信状态,信盘面**:ah 的 status/title、job 的 COMPLETED、agent 的 IDLE 都可能撒谎。每次判断都物理验证:capture-pane 看真实内容、git diff/grep 看真落盘、ls/wc 验产出。agent 自报会撒谎(把 INVENTORY 标 ✅ 实则 0 落盘)。
2. **in-loop 盯到结果**:派活=你的任务开始。用后台轮询(job/agent 状态或 pane)盯到真出可信结果,不半途放手等通知。
3. **投递文本的安全铁律**:**绝不用 `printf`/`echo` 双引号传待发文本**——双引号里的反引号会被 shell 命令替换真执行(曾误启一整套 rogue 栈,险 OOM)。正确:写文件 → `tmux load-buffer` → `paste-buffer`;经 CLI 传用 `ah ask <id> "$(cat 文件)"`。
4. **派新任务前重置上下文**:agent 攒 token 到一定量(经验值:≥2 单未清),派**新**任务前先重置它的会话。**机械姿势**:`/clear` 这类会话命令不走 `ah ask`(那会创建一个 job),直接投 pane:`tmux -L <socket> send-keys -t <pane_id> '/clear' Enter`;pane_id 用 `tmux -L <socket> list-panes -a -F '#{session_name} #{pane_id}'` 现查,勿硬编码。铁律:**只清 IDLE agent**(`ah ps` 确认);清后等 pane 出现全新 CLI banner 再派单;**绝不对 BUSY agent 投任何键**(会把键混进它的工作流,实测出过误锁事故)。
5. **共享额度意识**:同 provider 的多个 agent(以及 operator 自己,如果同 provider)可能共用同一账号的用量池。池紧时:审阅少派一道、独立可并行的活直接 `ah ask <worker>` 绕开 master 省它的开销。
6. **直连 live 栈**:operator 的 `ah` CLI 默认可能指向别的 state dir。用 `AH_STATE_DIR=<live 栈路径> ah ps` / `ah ask` 直连在跑的 daemon。
7. **operator 只动脑不动手**:operator 通常跑在最贵/最强的模型上,它的价值是判断、拍板、审阅、盯梢——不是执行。凡是能写成 brief 派给 master/worker 的活(改代码、跑测试、写文档、清理树、取证 grep),一律下派;operator 亲自动手仅限三类:(a) 只有 operator 权限能做的(开 PR/合并/凭据操作/对 master 本体的注入),(b) 栈本身瘫痪、没有 worker 可派的应急恢复,(c) 一条命令能完成的只读核验(看盘面/读状态)。动手前自问:这活为什么不能派下去?
8. **称呼 agent 永远带 provider**:说 `a1-codex`、`a4-claude`,不说裸 `a1`/`a4`。裸 id 在多实例同 provider(如两个 codex)或换拓扑后必然搞混谁是谁;brief、报告、规则文档里一律 `id-provider` 全称。
9. **master 上下文压缩会吞消息**:master 接近 100% 上下文触发压缩时,发给它的消息可能无声丢失。给 master 派单/传话后,验证动作真发生(job 落库/回执出现),没发生就重发。
10. **派单竞态**:给「刚完成任务」的 agent 立刻派新单,prompt 可能永远没落 pane(job 卡 DISPATCHED 直到误判 STUCK)。等 agent 确认 IDLE 再派;重派 brief 必须自包含(新会话无前情)。处置:cancel job → kill agent → up → 等 IDLE → 重派。
11. **「内部后台任务」型假完成(部分 provider 的 harness 行为)**:有的 provider(实测:antigravity/Gemini 系)会把长命令跑成自己 harness 的内部后台任务,等待期间回合自然结束——编排层把「我在等编译」这种中间叙述闩成最终回复,job 假 COMPLETED 但零产物。这是 harness 行为,**规则文档写「禁后台、做到 commit」治不了**(实测 4/4 照犯),hook 层也堵不住(该 CLI 的 Stop hook 是异步发射后不管,实测无法阻断回合)。SOP:job 标完成先物理验证(commit/落盘);假完成=忽略状态、绝不重派(上下文完好,其内部任务完成会自动唤醒它续干),盯到真产出。根治只能在编排器侧做真完成信号检测。
12. **空闲 agent 的幽灵文本卡派单**:某些 CLI(实测:claude)空闲时 composer 会残留它自己上轮未提交的建议文本或推送 banner,被派单扫描器误判成待处理 prompt → agent 锁死、新 job 永不送达。可从偶发升级成「每单税」。处置:kill + up 重建该 agent + 自包含重派(排队 job 会被级联删,重派必须带全上下文);预防靠编排器侧忽略幽灵/banner。

## 五、工程纪律清单

这些是我们一路踩坑换来的硬纪律,派 worker 时带上,审阅时逼落实:

| 纪律 | 为什么 |
|---|---|
| **物理实证验证** | agent/subagent 会自欺或报空壳(claim 92% 实则 0 落盘);ls/wc/grep/跑测试强校验落盘与行为。 |
| **不信二次信号** | tmux title/wait/status 会撒谎;忙时也每 ~60s 亲自 capture-pane 看实际屏幕。 |
| **TDD 红绿** | 用测试把 brief 每条契约物理锁死,防实现漂移;按层(纯逻辑/集成/e2e)分严格度。 |
| **测试授权分层(考题独立、答题自由)** | 实施者自己出题自己答=自指测试:对任务理解错了,测试跟着错,红绿全绿也是自洽幻觉(实测:把"清洗子进程环境"理解成"清洗自己进程环境",no-op 修法带着全绿测试差点进主干)。**验收测试的断言目标(断言什么/在哪层)必须来自实施者之外**——brief 里写死,或安全关键任务由审计者先写验收测试;细粒度单测仍归实施者保 TDD 内环速度;审计逐任务必查"测试断言的是契约边界可观测行为还是实现内部状态",自指测试直接拒收。 |
| **串行构建防 OOM** | 资源受限环境多实例并行 cargo 会 OOM 杀主控;串行跑(`CARGO_BUILD_JOBS=1` + `--test-threads=1`)。但 CI 默认并行绿才是并发安全真验收。 |
| **本机测试面收窄 + CI 唯一实证门** | 会拉起 daemon/tmux 的集成与 e2e 测试在开发机上跑,曾多次误杀在跑的编排栈;本机只跑纯逻辑单测+编译级校验,全量实证交给 CI 隔离 runner。 |
| **baseline 对照抓真回归** | 别放过「红灯说无关」;对照主干单跑,证明是既有还是本次引入。 |
| **删/改公共符号跑全量** | 只测局部会漏掉别处调用点;至少全量编译确认所有引用。 |
| **改契约测试先出断言表** | 动契约测试前,「每条 assert(行号)↔ 契约位置」对照表先交审;表只证一致性,没钉死的行为拿主干代码做行为锚。防「改一处过一处」的打地鼠。 |
| **接 brief 先质疑假设** | zoom-out 四问(根本动机/契约一致/对齐/该不该 push back),技术对了但方向错=白干。 |
| **复合任务必拆分** | 「收集精确证据」(严谨,派实现者)和「基于证据做判断」(分析,派设计者)不交给同一个 agent。 |
| **worker 角色每次兜底** | agent 空白沙箱里易漂移越权当 PM;规则文档定死身份外,派单时 brief 头部再重申「你是 worker,不自派单/不做 PM」。 |
| **验证档案 fill-once** | 编译/测试/验收命令在接入时一次性沉淀进 `VERIFY.md`,agent 查表执行不现场重导;否则每个任务都重想一遍,推导错误(跑错测试包/漏约束参数)反复发生。 |
| **定向迭代/全量收口** | 红绿循环中间只跑受影响的定向测试(按文件/模块,不按名瞎过滤);全量串行只在收口跑一次作交付门。中间反复全量 = 每轮白烧数分钟编译(实测一单 45 分钟里编译占大半)。 |
| **审阅分配不吃并发通道** | 同 provider 多实例(如 a1-codex + a2-codex)的第二实例是为**并行实施**加的,不是专职审阅员。审阅机会主义分配:有并行实施任务排队时,审阅交给其它角色(如 a4-claude)或等空闲,绝不占用第二实施通道;仅当该实例真空闲时才顺手接审。铁律只有一条:**不许同实例自审**(独立上下文才有审的价值)。 |
| **回滚自检验测试真守护** | 「断言正确 + 全绿」仍可能是空转测试:断言锚在别处早已修好的代码上,根本没穿过本次 diff——回滚本次改动它照样绿=零回归保护(实测:测试直接调共享 helper 而非被修的路由,两轮才发现)。验法:临时回滚 diff,测试必须变红;审计者独立复现这一步。修法优先开「生产与测试共用的薄缝」让测试穿过真实路径。 |
| **可行性结论必须实测定谳** | 文档考古得出的「可行」可能借了同名框架在**别家 CLI** 里的语义(实测:hook 阻断方案文档层 FULLY FEASIBLE,隔离沙箱实测 CLI 对 hook 发射后不管、stdout 从不读取,NOT VIABLE 反转)。机制断言不实测不采信;要依赖某机制前,先花一个小时级 spike 在隔离环境拿一手证据。 |

## 六、zoom-out 四问(高风险/含糊任务开工前)

1. 真正要的用户结果是什么?
2. 哪条假设可能是错的?
3. 什么证据能证伪「成功」?
4. 最小的安全下一步是什么?
