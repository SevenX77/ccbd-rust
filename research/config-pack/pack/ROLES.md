# 角色原型 → provider 能力(参考表)

下面是我们自托管开发 ah 一路实测出的角色原型和它们适配的 provider 能力。**这是参考,不是强制**:你可以只用其中几个、合并、或按自己项目重定义。每种角色几个实例、配什么 provider、`id` 叫什么,都由你在 `ah.toml` 里定,并在对应 `.ah/rules/<id>.md` 的「角色定位」里写清。

| 角色原型 | 干什么 | 推荐 provider 能力 | 实测适配依据 |
|---|---|---|---|
| **主控 / 协调者 (Coordinator)** | 规划、错峰排期、分派、设计辩论、审阅、收敛;物理实证;跑目标闭环 | 强语义 + 长上下文 | 要读长 diff/日志、统筹多 agent、和设计者带论据辩论,弱推理会漏判 |
| **实现者 (Implementer)** | 主力编程 + 测试 + 调试 + TDD;**也承担严审**(审别的实例的产出);**小机械设计**直接归它 | 强机械精确、严谨 file:line、不发散 | codex 类在「精确改一处 + 全覆盖测试」上稳;推理型易发散越界。多实例=同一角色为并发,规则文件内容相同 |
| **设计者 (Designer)** | 架构级设计主笔 + 第一性 research + 只读 sweep 测绘;**spec 忠实度对抗审** | 发散思维、第一性原理、看长远不打补丁 | 发散模型天然挑战前提、推翻问法,补丁型模型(claude/codex 类)做不到;它推理出的架构缺口多次赢过主控的印象(孤儿进程缺口实锤)。**弱点:讨好型+会自说自话实施,须配护栏(见 design.md)** |
| **审计者 (Auditor)** | e2e、深度语义审计、独立二审 | 强语义理解 + 集成能力 | 复合语义/集成场景;作为实现的独立第二双眼 |

## 设计权在谁手里(本包最重要的一条角色划分)

**架构级课题的设计主笔 = 设计者(发散型),不是实现者也不是主控。** 但设计者的产出不直接生效——要经 **主控带事实论据辩论到收敛**(流程见 GUIDE「设计管线」),再由实现者转写成严谨 spec。分工依据:

- 发散型主笔**设计**(比较优势:第一性、发散、看得远);
- 机械精确型主笔 **spec**(比较优势:严谨、file:line、TDD 框线)——设计≠spec,是两个产物,不可由同一比较优势的角色包办;
- **实测反例**:让设计者主笔 spec,它在最危险的决定上写了一句错误的安全断言,当晚兑现成活栈事故;让实现者主笔架构设计,得到的是补丁不是架构。

**小机械设计例外**:参数化一个值、改一处契约这类聚焦小设计,直接给实现者,比给发散型稳(后者易过度扩 scope)。

## 执笔权(闸门产物归 fail-safe 一族)

失败模式不对称:严谨型(claude/codex 类)错在保守(误拒收,fail-safe),发散型错在自信编造直接进产物(fail-dangerous)。所以 **一切闸门产物——tasks.md、TDD 框线、验收测试代码、spec 硬流程——执笔权只在严谨 agent**;发散型只有辩论席位与实施位。

- **验收测试执笔顺位**:实现者是发散型时,验收测试代码由**审计者先写 RED 并 commit**,实现者纯实施变绿、**不得增删改测试文件**;主控在 brief 里钉测试名+断言目标是合规上限,让实施者自己写验收测试=实施者自证,违规。claude 兜底顺位:**审计者优先于主控**(主控执笔是最后手段)。
- 审计者审实施不审自己写的测试;测试本体由主控亲验 + CI 把关。**唯一铁律不变:不许同实例自审。**
- 实施者的细粒度内部单测可自写,但不算验收证据。

## 怎么用这张表

- **拆分复合任务**:别把「收集精确证据」和「基于证据做抽象判断」交给同一个 agent。让实现者(严谨)先落证据,设计者(分析)再推理评估。
- **多实例 = 并发,不 = 新角色**:两个实现者实例是为并行实施加的,规则文件内容相同;审阅在实例间交叉(唯一铁律:不许同实例自审),但不为审阅占死第二实施通道(见 GUIDE 纪律表)。
- **角色↔provider 是能力匹配,不是身份绑定**:换 provider 不换角色语义;但**设计者必须与主控异族**(不同模型家族)——同族发散=回音壁,无效。

## 落到配置

举例(仅示例,自行增删改),某项目可能声明:

```toml
[agents.impl]      provider = "codex"        # 实现者(实例1)
[agents.impl2]     provider = "codex"        # 实现者(实例2,同角色,纯为并发)
[agents.design]    provider = "antigravity"  # 设计者
[agents.audit]     provider = "claude"       # 审计者
```

然后 `.ah/rules/impl.md`(impl2.md 与它**内容相同**)、`design.md`、`audit.md` 各写自己的「角色定位」。
