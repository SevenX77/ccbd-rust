# 递归熵减飞轮：计算化软件工程的神经网络与自由能模型 (a3-flywheel-divergent.md)

本文件由 a3 worker (explore 角色) 独立撰写。我们放开具体的工程细节与清单，直接从第一性原理出发，试图为“编程开发场景”构造出一套极简、自指且可递归的“开发系统自优化飞轮”，并对其神经网络类比进行深度批判，引入更契合的物理世界模型。

## 一、 熵减飞轮的最简数学表达：神经网络类比

我们可以将“基于 Spec 的开发系统”强行映射为一台正在进行“反向传播训练”的神经网络模型。在这套类比中，核心概念的对应关系如下：

### 1. 损失函数 (Loss Function)：语义偏离度
项目的熵（混沌度）即为系统的 **Loss**。它衡量的是**“人类意图（设计/Spec）”与“机器物理行为（代码执行/编译）”之间的语义空间距离**：
$$Loss = H(Spec \mid RunningCode)$$

在实际工程中，这个 Loss 是离散的，由以下代理指标（Proxy Indicators）逼近相加：
* **编译/类型校验噪声**（类型不匹配的硬报错数）；
* **功能断言偏差**（TDD 失败测试用例的占比与重置量）；
* **一致性漂移度**（文档与代码符号、API 架构契约之间的死链/脱节数）。
当所有测试变绿、没有类型错误、契约与代码完全对齐时，$Loss = 0$。

### 2. 权重矩阵 (Weight Matrix $W$)：知识库与契约（KB）
神经网络的权重 $W$ 代表了系统已学会的知识。在我们的 SCS 开发飞轮中，**知识库（Knowledge Base）——包括 `AGENTS.md` 规范、`.ah/rules/` 规则、`FRONTEND_UI_SPEC.md` 样式规矩、以及沉淀的 `skills/` 工具——就是系统的权重 $W$。
权重 $W$ 决定了 Agent 在面对输入需求时，动作空间（Action Space）的概率分布与收敛速度。

### 3. 反向传播与梯度更新 ($\nabla W$)：摩擦力反弹与规则写入
* **梯度（Gradient）** 代表了系统当前的“阻力与纠错方向”。在开发中，它表现为 Worker 在执行任务时**撞墙的频率与返工摩擦力**（例如：AI 试图在组件里写 raw fetch 被门禁拦截，或者多次编译因 OOM 崩溃）。
* **参数更新（Optimizer Step）** 则是系统根据这个“摩擦力梯度”，**反向重构和收紧知识库（KB）**的过程。
  $$W_{t+1} = W_t - \eta \cdot \nabla W$$
  当 Worker 反复在某个地方写错（梯度大），Master/SCS 就会在此处为 KB 新增一条硬规则（如“沉淀为 Skill”或“加入硬性 pre-commit 钩子”），从而修改了权重 $W$。

## 二、 归因问题：智能体贡献度（Credit Assignment）怎么算？

在神经网络中，每个神经元的贡献是通过偏导数（梯度）来分配的。在多智能体（Master+Workers）团队中，每个 Agent 到底减少了多少熵？

如果我们只看表面的 $Loss$ 曲线，会陷入一个致命的归因陷阱：
* **情境**：Worker `a1` 写了一段新代码，由于没有写测试，此时系统没有暴露任何错误，$Loss$ 表面上降为了 0。接着，Auditor `a4` 进场，编写了 10 个边缘测试用例，瞬间引爆了代码中的 5 个 bug，导致 $Loss$（测试失败数）暴涨。
* **错误归因**：`a1` 减了熵，得高分；`a4` 增加了系统的熵（把项目弄红了），扣分。

### 正确的归因计算法：显性熵与隐性熵的转化
我们必须将项目的熵分为**显性熵（Explicit Entropy - 已知的红灯/报错）**与**隐性熵（Implicit Entropy - 未知的缺陷/假阳性绿灯）**：
$$Loss_{total} = Loss_{explicit} + Loss_{implicit}$$

* **Auditor（审计者）的贡献**：在于**“将隐性熵转化为显性熵”**。虽然它让红灯变多了，但它降低了系统的“不确定性（真隐性熵）”。因此，它的贡献度等于其暴露出的失效断言密度。
* **Implementer（实现者）的贡献**：在于**“消灭显性熵”**。即把已知的红灯跑绿，其贡献度等于其修复的断言数量。
* **SCS 级系统** 通过这种“分流归因”，能客观计算出：在一次 PR 循环中，究竟是实施者拿到了消灭显性熵的 Credit，还是审计者拿到了发掘隐性熵的 Credit。

## 三、 类比的裂缝：神经网络隐喻在哪里会骗人？

神经网络比喻非常精美，但在指导软件工程时，它存在三处致命的物理断裂：

1. **梯度的连续性 vs 软件的离散悬崖**：
   神经网络依靠“微小变化产生微小影响”的连续凸空间。但在代码世界里，**软件是离散且非线性的**。改动一个字符（将 `==` 改成 `!=`），系统的编译状态会发生从 0 到 1 的断崖式坍塌。软件没有“平滑的斜坡”供你慢慢滑向最优解，它是一个由悬崖和深谷组成的碎形空间。因此，AI 无法通过“微调代码”来逐步逼近最优解，它必须进行**语义跳跃（Semantic Jumps）**。
2. **静态权重 vs 解释器的二阶熵增**：
   神经网络中的权重 $W$ 被激活时，是确定性的数学矩阵乘法，不会引入新噪声。但软件的“权重”（即 `CLAUDE.md` 等自然语言规则）在生效时，**必须被另一个 LLM 读取并解释**。这意味着，我们的规则本身也是带噪的，读取规则的过程会产生**二阶熵增**。规则越多，大模型读规则的开销越大，失焦的概率越高。
3. **过拟合与防御性退化（Overfitting）**：
   如果开发系统无节制地根据开发摩擦力自动更新规则，它会陷入“过度防御”。系统会生成成百上千条极其琐碎的 lint 规则和流程限制，把开发者的手脚完全捆死，导致开发效能归零。

## 四、 更本质的表述：自由能最小化与主动推理（Active Inference）

相比于神经网络，**卡尔·弗里斯顿（Karl Friston）的自由能原理（Free Energy Principle）**能够更本质、更自洽地解释这台开发飞轮的运行逻辑。

### 1. 软件工程的自由能公式
一个开发中的项目可以被看作是一个试图在多变环境中维持生存（编译通过、功能正确）的生命体。
* **内部模型 (Generative Model)**：即我们的 Spec 文档和设计蓝图。
* **外部世界 (Environment)**：即真实的物理运行环境（CPU 状态、浏览器渲染、测试断言）。
* **变分自由能 (Variational Free Energy)**：即设计（Spec）与代码物理行为之间的偏离度。

根据自由能原理，系统必须通过以下两种路径来最小化自由能：

* **路径 A：感知推理（Perceptual Inference - 更新设计）**：
  当物理测试跑不过，且确认是原本的设计 spec 不合理、脱离实际时，**系统必须修改内部模型（Spec/设计）以对齐物理现实**。这就是为什么我们强调“设计页不是死文档，必须随迭代回写更新”。
* **路径 B：主动推理（Active Inference - 更改物理）**：
  当设计 spec 是对的，但代码跑错了，**Agent 必须对外部世界采取行动（改写代码）**，使物理编译和测试结果收敛，对齐内部模型。

### 2. 递归飞轮的自指圈闭合
这台自由能飞轮的递归机制是自持且不发散的，因为**Spec（内部模型）与代码行为（外部世界）在通过双向的主动/感知推理进行动态对齐**：

```
     人类意图 (高维高熵)
           ↓
    [ 内部模型 (Spec) ]   ← (感知推理: 发现事实不符, 更新 Spec)
           ↓↑ (自由能: 偏离度)
    [ 外部物理 (代码运行) ] ← (主动推理: 编写代码, 跑通测试)
```

这种自指圈之所以能收敛，是因为它的底座牢牢钉在**“真实的物理执行（Tests/Compiler）”**上。外部物理（硅片上的代码执行）是一块永远不会说谎的终极重力场。任何漂移和幻觉在碰到这块重力场时，都会转化为可以被测量的变分自由能（报错信号），从而强迫系统通过更新代码或修正设计来实现自愈。
