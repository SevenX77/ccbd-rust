【调研两个真实问题,各自回答四问】

**问题一:服务端状态访问 + 事件驱动缓存失效 + UI 副作用边界(State sync & cache invalidation policy)**
用户为怕数据不同步,选了"严格一刀切"策略(大概是每次改动就全量刷/全局失效),但不知道业界怎么做到"既保证数据不漂移、又性能好"。
- ① 业界这类问题的**成名标准解法**是什么?(点名具体模式/库:如 server-state 管理、stale-while-revalidate、按 cache-key 粒度失效、optimistic update 等——用大白话解释每个是干嘛的)
- ② "一刀切全量失效" vs "按 key 粒度失效"的真实取舍(同步安全 / 性能 / 复杂度)
- ③ 用户已经有 SWR,标准做法下 SWR 该怎么用才对(cache key 设计、mutate/invalidate 粒度、什么时候 revalidate)
- ④ 这条架构规范若要写进一个"可复用配置包"并被**强制执行**,该长什么样(一句话铁律 + 可机器检查的门禁点,例:禁止在 SWR 层之外手搓 fetch)

**问题二:单一真相源 + 多组件投影(Single-source compile result + event-driven projection)**
用户的设计意图:一个后端接口出 compile 结果 → 前端多个组件投影同一份结果、各放各的位置 → 每次改文件触发全局 compile 事件。实际病症:agent 常在某个组件里就地打补丁(甚至把 compile 逻辑写进前端组件),只有那一个组件亮错、不走统一事件;而且到处发不必要的请求导致 app 越来越卡。
- ① 这个"一份真相源 + 组件订阅投影 + 事件驱动更新"是不是业界成名模式?点名它(single source of truth / derived-state selector / event-driven invalidation 等),大白话解释
- ② 用户纠结的两个点业界怎么答:**需要触发全局 compile 吗**?**怎么更新数据不影响性能**?(coalesce/debounce、增量编译、订阅切片 selector 等)
- ③ "组件各自算自己的 compile / 各自发请求"这种反模式,业界怎么从结构上杜绝(不是靠审阅,靠架构+门禁)
- ④ 同问题一④:写进可复用配置包 + 强制执行该长什么样(铁律 + 机器可检门禁点)
