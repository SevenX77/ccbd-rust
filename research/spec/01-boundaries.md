# 阶段一：宏观定位与边界讨论 (Boundary & Positioning)

> **目标**：明确 ccbd-rust 在 spec-coding 工具链中的物理与逻辑边界，消除 L2（调度层）与 L3（编排层）之间的功能重叠，确立系统的「不做之事」清单。
>
> **完成判据**：
> 1. 主控端与分析师端对「ccbd-rust 绝对不做之事」达成书面共识。
> 2. 确定新旧 CCB 系统的过渡与替代策略。
> 3. 定义 L3 接口的稳定性承诺（Stability Guarantees）。

---

## 1. 讨论议程 (Agenda)

### 议题 1：Spec-Coding 工具栈中的生态位定义
*   **背景说明**：当前存在 `superpowers` (skill 包)、`Kiro` (方法论/IDE)、`Claude Code` (执行端) 等多个层次。ccbd-rust 应当定义自己是这些工具的「通用 PTY 承载层」还是专门为「ccb 体系定制的后端」。
*   **讨论要点**：
    - ccbd-rust 是否支持非 ccb 相关的通用 PTY 调度？
    - 它与 IDE (如 VSCode Companion) 的数据交换边界在哪里？
*   **产出形式**：一张描述 ccbd-rust 在工具栈中纵向位置的架构层级图（描述文字）。

### 议题 1 最终决议
*   **决策点 A 决议**：A1（L2 必须理解 Agent 协议）。ccbd-rust 在底层集中处理 VT100 解析并捕捉空闲（Idle）标志，以唯一事实形式向外暴露状态，避免多客户端重复解析导致的状态撕裂。
*   **决策点 B 决议**：B1（L2 拒接非 Agent 进程）。贯彻 YAGNI 原则，坚决不通过后门接管 `cargo build` 等普通 Bash 任务，避免导致生命周期和状态机的 Scope Creep。
*   **补充决议 (2026-04-26)**：针对识别失效场景，建立 7 步反馈闭环（见 issue-1b）。确认 L3 LLM 兜底断言仅作为临时逃生舱，系统必须通过 Evidence 采集与自动化测试，最终将异常场景固化回 L2 的 Rust 原生规则中。
*   **决议日期**：2026-04-26
*   **达成方式**：Master Claude 与 Gemini 双方共识收敛（经用户授权）。

### 议题 2：ccbd-rust 「绝对不做之事」清单 (Anti-Goals)
*   **背景说明**：为了保持 L2 层的极简与高可靠，必须明确哪些逻辑属于 L3（Master Claude）的范畴。
*   **初始候选清单**（需讨论扩展至 5 条以上）：
    1. **不解析 Prompt**：绝对不感知用户发送给 Agent 的自然语言内容。
    2. **不管理多轮对话上下文**：不负责存储或拼接对话历史。
    3. **不执行 LLM 推理**：本身不直接调用任何 LLM 接口。
    4. **不进行业务级重试**：当 Agent 逻辑报错时，L2 只汇报状态，不自动重试业务。
    5. **不管理 API Keys**：LLM 访问凭证由 Agent CLI 自己管理或 L3 提供，L2 仅负责进程环境。
*   **产出形式**：一份经过终审的「禁止清单」。

### 议题 2 最终决议
*   **最终决议结果**：确认完整的 **10 条 Anti-Goals 清单**（包含上述 5 条初始清单，并新增以下 5 条）。
    6. **不做编排逻辑**：不感知任务依赖图、串行/并行控制等工作流逻辑，L2 不知晓 `agent.send` 之间的上下文依赖。
    7. **不做权限鉴权决策**：无 User/Role 体系，纯依赖操作系统底层的文件权限（Socket 的 `chmod 0600`）执行硬隔离。
    8. **不做长期对话归档**：不存储长期日志用于历史查询，事件溯源（Event Sourcing）仅作为近期断点续传支撑，历史查询由 L3 直读 Provider JSON。
    9. **不做内容审查 / 安全过滤**：不对 L3 传入的 Prompt 执行注入检测、密钥剥离等合规检查，完全无条件透传。
    10. **不做成本/计费追踪（Gemini 新增）**：不解析计费 Token 数，不关联任何 API 消费金额预测，财务指标必须由上层解析。
*   **决议日期**：2026-04-26
*   **达成方式**：Master Claude 与 Gemini 双方共识收敛（经用户授权）。

### 议题 3：与原版 Python CCB 的边界继承关系
*   **背景说明**：新系统是完全的物理替代，还是在逻辑上兼容旧的目录结构与配置？
*   **讨论要点**：
    - 是否需要「双向运行」过渡期？
    - 数据库 (SQLite) 结构是否需要考虑旧版数据迁移？
*   **产出形式**：一份明确的「新旧替代路线图」。

### 议题 3 最终决议
*   **决议结果**：执行「断代式完全替代，但保持接口兼容」的策略，具体包含：
    1. **不共存**：Rust Daemon 启动时要求旧 Python 进程必须停止，不允许在同一工作区抢占。
    2. **配平兼容**：无缝读取原版 `.ccb/ccb.config` 格式，用户配置零修改。
    3. **拒绝数据迁移**：不自动将旧的 JSON 任务转换为 SQLite 格式，但系统会在启动时输出明确的 Warning，提示用户自行清理残余 mailbox 文件。
    4. **Systemd 隔离**：使用独立的 `.service` 名字（如 `ccbd-rust.service`）确保系统管理命令不冲突。
    5. **单二进制多模式**：采用统一的二进制产物（包含守护进程模式与兼容 CLI 前端），提供 `ccb ask` 等与旧版行为完全对齐的入口，简化分发。
*   **决议日期**：2026-04-26
*   **达成方式**：Master Claude 与 Gemini 双方共识收敛（经用户授权）。

### 议题 4：L3 接口语义与稳定性承诺
*   **背景说明**：针对近期发生的 tmux pane 重分配导致的投递失败事件，定义 L3 观察到的稳定接口。
*   **讨论要点**：
    - **ID 稳定性**：如何确保 `agent_id` 在后端重启后依然有效？
    - **故障透明度**：当物理层（如 tmux）失效时，L2 如何即时「告知」而非「欺骗」L3？
*   **产出形式**：L3 通信契约中关于「状态一致性」的文字承诺。

### 议题 4 最终决议
*   **决议结果**：确立 7 条 L3 接口稳定性核心需求作为后续阶段三协议设计的刚性边界。
    - L2 负责屏蔽底层设施的抖动（通过 **R-DISPATCH-1**, **R-RECONNECT-1**, **R-IDEMPOTENCY-1**）。
    - 故障和系统状态必须 100% 透明和强类型化，禁止黑盒（通过 **R-DISPATCH-2**, **R-OBSERVABILITY-1**, **R-ERROR-CODES-1**）。
    - 通信协议本身必须具备演进的可预测性（通过 **R-API-COMPAT-1**）。
*   **决议日期**：2026-04-26
*   **达成方式**：Master Claude 与 Gemini 双方共识收敛。

### 议题 5：资源与沙盒的所有权边界
*   **背景说明**：bwrap 沙盒创建的文件、日志文件、临时 Socket，其生命周期谁来管？
*   **讨论要点**：
    - 谁负责清理僵尸资源？
    - L3 是否有权直接访问 L2 的沙盒内部文件？
*   **产出形式**：资源所有权分配表。

### 议题 5 最终决议
*   **决议结果**：建立清晰的资源矩阵，核心原则为「L2 资源闭环，外部资源敬畏」。
    1. **内部资源 (DB, Socket, Sandbox Dir)**：L2 拥有绝对所有权，负责创建、状态转换与清理；L3 仅能通过 RPC 间接访问。
    2. **外部资源 (Git Repo, Config Files)**：归用户/L3 所有。L2 必须以 **只读 (Read-Only)** 形式读取配置文件，且不对 Git Repo 执行生命周期管理。
    3. **清理机制**：采取 Startup Reconcile 为主、Systemd BindsTo 为辅的幂等清理逻辑，确保无孤儿资源残留。
    4. **隐私约束**：stderr 日志严禁记录对话内容，仅保留审计事件。
*   **决议日期**：2026-04-26
*   **达成方式**：Master Claude 与 Gemini 双方共识收敛。
