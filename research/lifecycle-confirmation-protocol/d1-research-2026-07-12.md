## 一、Claude worker 侧 `UserPromptSubmit`(UPS)

### 已实证(读代码)
1. **UPS 注入被刻意角色门控到 Master,worker 只注入 Stop**——`home_layout.rs:228-233`:
   ```
   if let Some(ctx) = active_hook_push_ctx(hook_push_ctx, "claude") {
       if role == HomeLayoutRole::Master { hook_specs.push(materialized_ah_hook(ctx, "UserPromptSubmit")); }
       hook_specs.push(materialized_ah_hook(ctx, "Stop"));
   }
   ```
   即 worker 拿不到 UPS **不是"没写"而是"被 `if role == Master` 一行拦掉"**。o1 的"worker 未注入 UPS"结论成立,但机理更精确。
2. **hook 命令构造器是事件无关的**——`build_ah_hook_command(ctx, event)`(`home_layout.rs:674-690`)对任意 `event` 生成 `ah agent notify --event <event>`。故给 worker 加 UPS **是一行去门控**(把 `:229` 的 `if role==Master` 放开),无需新造机制。
3. **Master 侧 UPS 已被消费并驱动状态**——master notify handler(`agent.rs:987-989`)`"userpromptsubmit" => ("BUSY", false)`、`"stop" => ("IDLE", true)`;测试 `claude_master_hook_push_installs_ups_and_stop_with_master_sentinel`(`home_layout.rs:2367`)。
4. **【关键·推翻 o1 前提】worker 与 master 启动/投递机制完全相同**:
   - worker 以**裸交互式 `claude`** 在 tmux pane 里拉起:`agent.rs:275` `spawn_window(...)`;命令数组是 `["claude", "--dangerously-skip-permissions"]` / `["claude"]`(`agent.rs:684/:694/:714/:724/:738/:766`)——**没有把 prompt 作为命令行参数**。
   - prompt 通过**粘贴进 composer + 回车**投递:`agent_io/writer.rs:20-36`(`paste_buffer` + `press_enter_after_paste`),`agent.rs:1183` `should_press_enter_after_paste(&agent.provider, text)`。
   - 这与 master 收 prompt 的方式**逐字相同**(交互 pane + 粘贴 + Enter)。
   - **o1 说"worker 通过 `claude "prompt"` 非交互式执行,需实证命令行传参启动时 UPS 是否触发"——此前提为假**:worker 根本不是 one-shot 传参,而是交互式 pane 收粘贴 prompt,和 UPS 会触发的 master 路径同构。

### 裁决与置信度
- claude worker UPS **不是"理论可行未实证"(o1 措辞),而是"机制与已工作的 master 路径同构,仅差一行去门控"**——置信度显著高于 o1 给的。
- **待验-1(需 live 运行,本轮未覆盖)**:去门控后,claude 是否对"粘贴+回车"的 worker prompt 真的**触发** UPS hook。高先验(master 同机制已触发),但未经真实 worker session 观察前不宣布 100%。**这是 claude 侧唯一 待验。**
- 旁证:claude 版本 2.1.183(`research.md:86`);claude hook 引擎在本仓已被证明能路由非 Stop 事件——repo 测试覆盖 `PreToolUse` + `permissionDecision` 输出(`research.md:86`,`tests/pr4c_hooks_plugins.rs`、`tests/pr1b_readfirst_hook.rs`)。故"claude 能路由 UPS 这类事件"本身不是问题,只剩"worker 形态下 runtime 是否真发"这一步 live 确认。
- **备选载体 `SessionStart`**(o1 提):worker 生命期只跑一次任务,SessionStart≈任务开始,可作 UPS 的兜底载体;但同样 待验其在 worker 交互 pane 下的触发,归入 待验-1 的同类 live 观察。

### 已实证(d1 亲跑 strings + research.md)
- **agy 二进制确含 Pre/PostInvocation、Pre/PostTool hook 符号**——d1 本轮亲跑 `strings /home/sevenx/.local/bin/agy` 命中:
  - `agent.PreInvocationHook` / `agent.PostInvocationHook` / `agent.PreToolHook` / `agent.PostToolHook`
  - **`registry.NewPreInvocationHookFn` / `NewPostInvocationHookFn` / `NewPreToolHookFn`**(registry 构造器——比纯 proto 更强的"可路由"信号)
  - `hooks_pb.PreInvocationHookArgsH` / `PostInvocationHookArgsH`;`pretoolhooks.PreToolHookContext`
  - **嵌入的文档字符串 `### 3. \`PreInvocation\` Contract` / `### 4. \`PostInvocation\` Contract`**(二进制内置契约文档,暗示这是有明确契约的可配置 hook,非纯内部)
- **research.md:87 独立佐证**(此前一手 strings):`GetPreInvocationHookArgs`、`CustomAgentConfig.GetPreInvocationHooks`、`DeclarativeMixinConfig.GetPreToolHooks`、`jsonhook.JSONHookSpec.IsEnabled`、`jsonhook.ParseHooksFile`。

### 对 o1 引用的一处更正(重要)
- **o1 把 PreInvocation 存在性归给 `antigravity-hooks-preverify.md`——但那份 preverify 的 §3 只确认了 Stop 家族**。其 strings 命令(preverify `:184-185`)的搜索 pattern **包含** `PreInvocationHookArgs`/`PreToolHookArgs`,但其 §3 写出来的命中清单**只列了 `StopHook*`**,未把 PreInvocation 列为已命中。故 **preverify 不是 PreInvocation 的证据源**;真正证据在 **`research.md:87` + d1 本轮亲跑**。o1 结论(符号存在)对,但引错了文档。

### 关键 待验:符号存在 ≠ hooks.json 可路由
- agy 疑似有**两个 hook 配置面**:
  1. **`jsonhook` / `hooks.json`**——ah 实际注入的面(`~/.gemini/config/hooks.json`),**已确认可路由 `Stop`**(preverify §0/§3;ah 只注入 Stop,见 `inject_antigravity_hook_push:408-428`)。
  2. **`CustomAgentConfig.GetPreInvocationHooks` / `DeclarativeMixinConfig`**——proto/声明式 agent 配置面,**ah 目前不写**。
- 符号(尤其 `CustomAgentConfig.GetPreInvocationHooks`)证明**能力存在于某个面**;但**不证明 ah 注入的 `hooks.json` 面暴露 PreInvocation 事件键**。Stop 的 JSON key 是 `"Stop"`(对内部符号 `StopHook`);PreInvocation 的 JSON key 是否为 `"PreInvocation"`、`hooks.json` 引擎是否路由它、runtime 是否在任务开始真触发——**均未确认**。
- **待验-2(需 live agy session,本轮未覆盖)**:agy 的 `hooks.json` 面是否暴露并路由某个"任务开始"事件键(PreInvocation/PreTool),其**精确事件键字符串**,以及它是否在**任务开始时真触发**。符号存在使其**可能**,但不足以写成可实施契约。**这是 agy 侧的 待验。**
- 兜底(o1 提,采纳为设计输入):若 hooks.json 面不路由 PreInvocation → 走 wrapper 造 `worker_process_launched`/`started` 弱信号(与 codex 同路,见 §三)。

## 三、Codex hook 事件全集(汇总既有调研,不重复调研)

### 已实证(research.md,一手)
- codex-cli **0.135.0**(`research.md:85`)。
- **对本机 `codex.js` 跑 `strings ... | rg -i 'hook|hooks|Stop|PreToolUse|...'` 零命中**(`research.md:85`)——二进制里**没有任何 hook 事件字符串**(连 Stop 都没有)。
- 隔离 HOME 无 `$HOME/.codex/hooks.json`;`.codex/config.toml` 只有 trusted project + tui 配置(`research.md:85`)。
- **ah 的 codex materialization 只注入用户配置的 `extensions.hooks`,不注入 ah 完成推送 hook**——`home_layout.rs:1065-1067`:`materialize_hooks(...&extensions.hooks)`,非空才 `merge_codex_hooks`;而 ah 的完成推送 Stop hook 只在 **claude** 路径注入(`:232`)。故 codex 默认**无任何 ah 托管生命周期 hook**。

### 二手参考(未实证)
- cmux 文档称其 setup 写 `codex→~/.codex/hooks.json`(`docs/competitive/cmux-vibeyard-borrowable-for-ah.md:50`)——**机制参考,非本机 0.135.0 实证**。

### 裁决
- **codex 无任何原生"任务开始"钩子**(强支持:二进制零 hook 字符串)。o1 结论成立且**更强**:连 codex **原生 Stop** 都未在本地版本实证(仅 cmux 声称),完成侧亦可能需 wrapper。
- **必造信号(采纳 o1)**:codex 必须 wrapper 插桩——包装脚本 exec `codex` 前经 `CCB_SOCKET` 发 `ah agent notify --event started` 人工补齐开始信号。**注:这是被动信号"源"(wrapper 自发上报),不是主动探测**,与本设计"仅被动"约束不冲突(见 design)。
- **待验-3(cmux prior-art,未本地实证)**:codex 0.135.0 是否经 `~/.codex/hooks.json` 支持原生 Stop。对**开始信号无影响**(codex 开始必走 wrapper),仅影响完成侧是否也须 wrapper 造 Stop。

## 四、病历 #4(agy 假 COMPLETED)实证 —— 支撑 cgroup populated AND-锁

- `antigravity-premature-completion.md:5`:a3(antigravity)e2e 后 ah 标 `COMPLETED`,但 `jobs.reply_text` 只有一句"等待后台 cargo test",真实结论(`5 passed` + 物化校验)只在 tmux pane、从未写进 job 记录。
- `:66-69`:agy 的终态 log 事件 `PLANNER_RESPONSE DONE` 是**provider 回合边界(F2),不是 shell/后台工作已结束的证明(F3)**;UI idle marker 只是输入状态行,"回到输入提示符"只证模型能收下一条 prompt,不证语义完成。
- **结论**:实证坐实 o1 病历#4——逻辑完成信号(Stop hook / PLANNER_RESPONSE DONE)只到 F2,必须与**物理 cgroup populated=0** 做 **AND 联锁**才达 F3。大模型自称完成不可信这条,根治靠物理联锁,不靠更多逻辑信号。
