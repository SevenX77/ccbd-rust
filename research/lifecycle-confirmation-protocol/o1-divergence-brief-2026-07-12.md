## 问题域(事实,不含结论,请自行判断)

### 事实一:任务开始信号缺失
当前 ah 的三个 provider(claude/codex/antigravity)在 hook 层面,**只有 Stop hook 落地**(任务结束时触发)。任务开始端(dispatch 之后、agent 真正开始处理之前)在 worker 侧**完全没有确认信号**——只有 master 才有 `UserPromptSubmit` hook。

代码实证(`src/rpc/handlers/agent.rs:877-901`):
```rust
if event != "stop" {
    tracing::info!(..., reason = "unsupported_worker_event", "ignored unsupported worker hook");
    return Ok(json!({ ..., "unsupported": true, "transitioned": false, ... }));
}
```
worker 发来的任何非 stop 事件,ahd 显式记录为"unsupported"并丢弃,不产生任何状态转移。

### 事实二:能力存量(不同 provider 已有的原始能力,供你判断谁能承载"开始信号")
- **claude**:已有 `UserPromptSubmit`/`PreToolUse` hook 事件类型,且有既存测试覆盖这些事件的解析路径(但如上所述,worker 侧收到后被丢弃,不是能力不存在,是接线没做)。
- **antigravity(agy)**:字符串层面存在 `PreToolHookArgs`/`PostToolHookArgs` 事件名(是否真的在这个 provider 的运行时里可靠触发、触发时机是否等价于"任务真的开始处理",未经实证)。
- **codex**:hook 事件全集**未经实证**,不知道 codex 是否有任何等价于"任务开始"的原生信号。

### 事实三:同族病历(这些不是孤立 bug,是重复出现的同一类问题——"没有 100% 可靠的生命周期信号,只能靠推断",据此推断出的状态经常是错的)
- **dispatch-ACK 竞态**:job 派发后,prompt 文本实际上没有落进 agent 的 pane/composer,但系统以为派发成功了,job 卡在假 STUCK 状态(本次会话里在 o1 agent 上亲历复现三次)。
- **claude banner/幽灵提示词误判**:CLI 启动横幅或残留的 composer 文本被误判为任务信号。
- **R2 就绪探针被自己的注入击穿**:曾经设计过一种"主动探测 agent 是否就绪"的机制,结果探测本身的注入动作把被测系统的状态搞乱了(后来改成被动读取语义 ACK 才修好——具体怎么改的,请自己去查,不在本单事实清单里提前告诉你结论)。
- **agy 假 COMPLETED**:antigravity agent 声称任务已完成(turn-end),但实际上根本没有真的完成或者根本没开始对应的产出。

**共同结构**:发送一个动作(派任务/清空输入框/点击/敲键)之后,系统**假定**它生效了,而不是**确认**它生效了。"发送 ≠ 生效"这个鸿沟,是本课题要解决的核心问题。

### 事实四:用户提出的抽象原则(这是用户的立场陈述,不是让你验证对错,而是让你围绕它展开设计空间探索)
用户认为:
- (a) 任务开始 hook 必须真正挂上,不能像现在这样打折成只有 Stop hook。"ahd 自己发的自己知道"这个理由不成立——幽灵提示词事故的根源正是发送任务后没有确认信号,系统以为发成功了但其实没有。
- (b) 生命周期确认的核心原则:推动生命周期前进的动作 = 一个 loop(while 循环)。跳出这个 loop 的条件 = 一个**明确的、代表生命周期状态改变的 100% 可靠信号**。例如:任务前 hook 发出的事件表明任务真的开始了 → "发送任务"这个 loop 的跳出条件达成 → 状态机推进到下一个 loop("等待对话结束")。
- (c) 当跳出条件为 false(还没达成)时,当前 loop 内部要做的"重试/纠正动作"本身也应该拆解成同构的、有自己独立确认信号的子 loop。用户给的例子:清空输入框这个动作 —— loop 条件 = 输入框是否已空;操作 = 按 ctrl+u,然后**观察 pane 里输入框的实际字符**确认是否真空了,如果还有残留(比如幽灵字符),再输入一个空格触发重绘,再次观察字符确认;确认输入框真空之后,才输入 job brief 文本,然后**再次观察 pane 字符**确认文本真的进去了;然后按 enter,**再观察字符**确认发送动作真的完成了(不是命令还停留在输入框里);发布动作完成后,进入下一个 loop——等待"任务已开始"信号,跳出条件是任务开始信号真的达成。
