## 〇、贯穿全设计的最硬约束:**区分状态层与动作层——状态层只用被动信号且禁 pane 造状态;动作层允许核验自己写入的回显**

**裁决:采纳 o1 病历#3 的治本方向,但必须把"状态层"与"动作层"分开(REV1 纠正原版把二者混成一条笼统禁令的概念错误)。** 这是本设计的第一约束,凌驾于所有 loop 形态之上。

### 三条互斥定性(决定什么被禁、什么被允许)

- **【状态层·禁】pane 文本内容造生命周期状态**:从 pane 文本推断"任务完成了/agent 活着/在忙"等**生命周期状态判定**。**继续禁,不动**(北极星冻结;三次幽灵事故的病根)。生命周期状态的唯一权威源是 hook(F2/F3/F4)+ cgroup/pidfd(F1),**pane 永远不做状态权威**。
- **【探测层·禁】注入探测字符去"问"状态**:注入 `echo $?`、DSR `\x1b[6n`、投试探键值主动问"你在吗/你完了吗"。**一律禁止**(病历#3 实证污染 composer;R2 探针自毁)。
- **【动作层·允许】核验自己刚执行的写入动作的回显**:清 composer 后**读回显确认它真空了**、贴 brief 后**读回显确认 pane 内容==要发的 brief**、按 Enter 后**读回显确认真发出去了**(composer 清空或内容进历史区)。**这不是探测、不是状态推断**——它不注入任何探测字符,只读"我刚做的那个写动作有没有生效"。**允许,且是必需的**(见下)。

### 为什么动作层核验必需(而非冗余可省)——用户实锤论证

**哪怕有 100% 可靠的开始信号,它也只证"某任务开始了",证不了"开始的是我要发的那个 brief"。** hook 信号对"我发的东西对不对"这一环**零覆盖能力**,失效场景 hook 永远察觉不到:

- composer 有上一轮残留文本,brief 追加其后 → 发出"残留+brief"混合体;
- Enter 被吞(没真提交)→ brief 停在输入框 → **开始信号永不产生 → 只等开始信号的系统永远傻等**(死锁);
- 以为 brief 打进去了其实没有(粘贴失败/时序错位)→ 按 Enter 发出空的或别的。

这些**不是"信号不可靠",是"我发对了没有"这一环**——必须靠**动作层核验**补上。它与"hook 作生命周期状态唯一权威"**完全不冲突,两码事**:动作层管"我发对了没",hook 管"任务真开始了没"。这就是"设计冗余"原则,而且这层冗余是**载重的**——它正是防止上面第②种"永远等不到开始信号"死锁的东西(见 §一:A 相位核验通过是 B 相位等待开始信号的前置条件)。

### 动作层核验与 hook double-check 的关系(用户裁决 ④)

- 生命周期状态仍以 **hook 为唯一权威**;pane 观察只作**辅助 double-check / 诊断入口**。
- **hook 信号与 pane 观察不匹配 → 触发错误流程,以 hook 为准**;**绝不允许用 pane 观察反过来否决 hook 判定**。

### 这条约束仍然限制了什么(诚实,不偷渡回主动探测)

- 仍**禁止**任何注入式主动探测(探测层永久禁)。
- 对无开始 hook 且无 wrapper 的 provider,**动作层核验能确认"我发出去了",但确认不了"模型真开工了"**(那要 hook)——此时开工侧只能等粗粒度被动存活(cgroup populated→1)或超时上抛(§三),**不退回注入探测**。动作层核验补的是派发正确性,不是开工确认;二者边界清楚,不越界。

## 一、LCP 四相位状态机(REV1:A 相位改为动作层核验 loop,非单次盲写)

REV1 前原版把 A 相位设计成"单次写、不核验就假定成功、确认权全交 B 的开始信号"——这落进了 §〇 论证的死锁(Enter 被吞 → brief 没发出 → 开始信号永不来 → 傻等)。**A 相位必须是逐步核验 loop:每一步先做写动作、再核验该动作的回显,核验不过则有界重试该原子步骤、仍失败则响亮 abort 上抛,绝不闷头进下一步。** 四相位:

```
[A 派发核验 loop]  每步"写→核验回显",不过则有界重试→仍失败则 abort+上抛(不进 B)
   ① ctrl+u 清 composer
   ② 核验回显:输入行确实空了      ─(不过)→ 重试①(有界)→ 耗尽→ abort+上抛
   ③ paste_buffer 贴 brief
   ④ 核验回显:pane 内容 == 要发的 brief ─(不过)→ 回①重清再贴(有界)→ 耗尽→ abort+上抛
   ⑤ 按一次 Enter
   ⑥ 核验回显:确实发出去了(composer 清空 / 内容进历史区)─(不过)→ 重试⑤(有界)→ 耗尽→ abort+上抛
   │  ⑥ 通过 → emit PerceptionEvent{dispatch_verified, epoch}(已确认"发对且发出")
   ▼
[B Start-Gate]──被动等开始信号(UPS/PreInvocation/wrapper-started)
      │  命中→ emit{started, epoch}→ Reconciler 置 BUSY@epoch
      │  超时(预算 T_start)→ fail-closed UNKNOWN + 上抛(§三)
      │  【关键】进入 B 时"brief 已发对且已发出"由 A⑥ 保证——B 只可能因"发对了却无开始 hook"超时(即 hook 通道问题),不再因"根本没发出"而死等
      ▼
[C Execution-Gate]──被动存活:cgroup populated=1 + PreTool/PostTool 事件
      │  T2 日志静默预算超 → 被动重读 cgroup populated → STALLED 响亮告警(不探测)
      ▼
[D Completion-Gate]──双锁 AND:①Stop hook(逻辑) AND ②cgroup populated=0 去抖(物理)
      │  两者皆满足 → job done
      │  仅①无② → "自称完成但后台仍活"告警(病历#4 根治)
      │  仅②无① → "退出未声明"看门狗(北极星 G2)
```

- **每相位的"动作/跳出信号/构造"**:
  - **A 派发核验 loop**:动作=清 composer(`ctrl+u`)/ `paste_buffer` brief / 一次 Enter,**每个写动作后核验其回显**(②验空/④验内容一致/⑥验已发出),复用 `agent_io/writer.rs` 现有粘贴通道 + capture-pane 读回显(**动作层核验,非状态推断,不注入探测字符**)。跳出=⑥核验通过。失败=任一步有界重试耗尽 → 响亮 abort + 上抛人/master,**不进 B**。
    - **实现纪律(补 o1 红队#2 异步渲染振荡)**:①核验读回显前给 PTY **settle 延迟 + backoff**,不 busy-loop;②**每次重试前先重清 composer(`ctrl+u`)再重贴**——使重试**幂等**,消除"上次贴其实落了但渲染滞后被误判 → 重贴成 briefbrief"的双贴危险;③核验是**与已知 ground truth(要发的 brief 精确文本 / 空)做等值比对**,不是开放式状态推断,渲染滞后只导致"不匹配→等一拍再看",不会误造状态。
  - **B Start-Gate**:动作=静默被动等。跳出=开始信号到达 `CCB_SOCKET`→ 写 PerceptionEvent。构造=事件驱动等待 + 超时 fail-closed(§三)。**因 A⑥ 已保证发出,B 不再需要"重派"来救"没发出"——见 §二层级论证。**
  - **C Execution-Gate**:动作=周期被动观测(cgroup populated + hook tool 事件)。跳出=进入 D 或降级。构造=Reconciler GC/心跳 pass 复用(对接 LF1 cgroup 委托),**无损、无注入**。
  - **D Completion-Gate**:动作=静默被动等双锁。跳出=①AND②。构造=对接 GF4 显式完成 + LF1 cgroup populated。

## 二、层级封顶:动作层核验环算哪一层(REV1 重新论证,仍守两层封顶)

**结论:动作层核验 loop 属【L1 原子动作级】,不是新增的第三层;o1 红队#3 的"最多两层嵌套"仍成立。**

- **L0 任务级**:A→B→C→D 相位推进。A 相位在 B 超时兜底重发时可被**横向再入**(状态机转移 B→A),这是**相位转移不是纵向嵌套**——任一时刻最多"处在某个 L0 相位内、其中跑着某个 L1 原子动作核验",不会出现 L0 里套 L0。
- **L1 原子动作级**:A 相位里每个原子写动作(清框/贴/Enter)**连同它的有界核验重试**,就是 o1 红队#3 命名的"原子动作级"。核验重试是**同一个原子动作的重做**,不是更深一层——它就是这一层本身。
- **为何不算第三层**:①核验读回显不再派生子环(它是"与 ground truth 等值比对"的单步判定,不是"为确认清屏又起一个 poll 子环去确认 tmux 命令有没有卡"那种递归);②核验失败的处置是"重试本原子步骤(有界)/ abort 上抛",**不下钻**。故动作层核验是 L1 的内容,不突破两层封顶。

**关于"重发/重派"归属的 REV1 更正**:原版把"重派"放在 B(Start-Gate 超时后重发)。REV1 后**A⑥ 已保证 brief 确实发出**,所以 B 超时**不再意味"可能没发出、需重发救"**——B 超时只意味"发对且发出了却没等到开始 hook"(hook 通道问题),对此**重发无用**,直接 fail-closed 上抛(§三)。真正的"发送正确性重试"回归到它该在的地方:**A 相位内各原子步骤的有界重试(L1)**。这比原版更干净:重试治"发送",fail-closed 治"hook 通道"。

| 相位 | 强信号/核验(首选) | 失败时处置 | 层级 |
|---|---|---|---|
| A 派发核验 | 每步动作层核验回显(②验空/④验一致/⑥验已发出) | 有界重试本原子步骤(每次先重清保幂等)→ 耗尽→ **响亮 abort + 上抛,不进 B** | L0 相位 + L1 原子核验 |
| B Start | UPS / PreInvocation(待验-2)/ wrapper-started | 超时→ **不重发(A 已保证发出)**、**不降级弱信号顶班** → fail-closed UNKNOWN + 上抛(§三) | L0 |
| C Exec | cgroup populated + PreTool/PostTool | tool 事件缺→仅 cgroup populated 兜底;日志静默→被动重读 cgroup→STALLED 告警 | L0 |
| D Completion | Stop hook AND cgroup populated=0 | 仅①/仅②→告警不判完成(§三);cgroup 缺→§五弱化路径 | L0 |

## 三、超时与 fail-closed 行为(明确裁决,不含糊)

### 裁决核心:开始/完成信号丢失超预算 = **fail-closed 到响亮 UNKNOWN + 上抛,绝不静默续跑,绝不主动探测**

- **A 派发核验失败**(REV1 新增,派发正确性的第一道):任一核验步(②/④/⑥)有界重试耗尽 → **响亮 abort + 上抛人/master,不进 B**。这拦住了"没发对/没发出却继续等开始信号"的死锁根源。
- **B Start-Gate 超时**(A 已保证发对且发出,故此超时语义收窄为"发出了却无开始 hook",o1 红队#1 的 silent hook failure):
  1. **不重发**(A⑥ 已确认发出,重发无意义且会重复投递);**不降级弱信号顶班**;
  2. → **判定 `UNKNOWN`,job 不进 BUSY、不派后续,上抛 master/人**;
  3. **弱被动信号(cgroup populated→1)只作告警的佐证**("brief 已确认发出、进程已起并耗资源,却无开始 hook——极可能 hook 通道损坏[权限漂移/socket 不可达],请人工核查"),**绝不用它替代权威把状态标 BUSY 续跑**。
  - **为何 fail-closed**:A 已排除"没发出";残留原因只剩 hook 通道损坏,重发无用。北极星铁律"watchdog=权威失败判决,不是退回低级信号的理由";用 cgroup"有进程在跑"当开工续跑=重犯"unknown 造状态"。故降级=响亮上抛,弱信号只佐证。
- **UNKNOWN 上抛后的诊断 double-check(用户裁决 ④)**:
  - 上抛后**允许 master 去看一眼 pane 做诊断性 double-check**——这是给 master 一个**诊断入口**弄清到底发生了什么(brief 是否真在 pane 里、有没有信任弹窗挡着等),**不是拿 pane 状态当权威判定**。
  - **hook 信号与 pane 观察若不匹配 → 触发错误流程,以 hook 为准**;**绝不允许用 pane 观察反过来否决 hook 判定**(pane 只辅助诊断,hook 恒为生命周期状态唯一权威)。
- **D Completion-Gate 超时/半锁**:
  - 仅逻辑①无物理②(自称完成但 populated≠0)→ **不判完成**,发"后台仍活"告警,继续等 populated=0(病历#4 根治);
  - 仅物理②无逻辑①(进程退了没 Stop 声明)→ **不静默判完成**,发"退出未声明"看门狗(北极星 G2),转 UNKNOWN 待人/master 裁;
  - 转 UNKNOWN 后同样适用上面的诊断 double-check(master 可看 pane 诊断,hook 与 pane 冲突以 hook 为准)。
- **预算参数**(方向,非冻结):T_start ≈ 10-15s(与 §四 Q2 T1 预算一族);A 各原子步核验的 settle 延迟/重试次数属实现调参;cgroup populated 去抖 300-500ms(北极星 2.1②);具体数值留冻结/放行后调,且可由 GF1 影子期实测。

## 四、与 GF1(事件脊柱)/ GF4(完成协议)的接缝

### 与 GF1:LCP 是 PerceptionEvent 的**投递源**,绝不自己写 DB
- LCP 采集的每个信号(`dispatch_verified`[A⑥ 核验通过]/`dispatch_abort`[A 核验失败上抛]/`started`/tool-event/`stop`/`cgroup_populated_flip`)都封为 **`PerceptionEvent`(带 `epoch=state_version`)**,经 MPSC 投递给唯一的 `StateReconciler`(GF1 单写)。**LCP 不调用任何 `mark_agent_*`**(GF1 P1 单写不变量)。动作层核验的通过/失败也是事件,统一进脊柱。
- 所有状态迁移(DISPATCH→BUSY→IDLE/UNKNOWN/STALLED)由 Reconciler 依事件 + epoch 单次 CAS 完成。**采纳 o1 §四.2**。
- **epoch 绑定根治病历#2(幽灵文本世代污染)**:每次任务绑定唯一 `state_version`;开始信号之前的一切 pane 字符被过滤。**且因 §〇 已删除 pane 感知,幽灵文本本就无法造状态——双重保险**。

### 与 GF4:LCP 的 Start-Gate 是 GF4 完成协议的**开工对称件**
- GF4 负责"如何显式收尾"(`ah job done` + Stop 强制),但它**假设任务已进 BUSY**。LCP 的 Start-Gate 提供那个"已真开工"的断言——**没有 LCP 开工确认,GF4 无法区分"任务真在跑等收尾"与"卡在派发通道从没开工"**。二者合起来才是完整生命周期括号。**采纳 o1 §四.1**。
- **D Completion-Gate 的双锁就是 GF4 完成机制的物理化**:①Stop hook = GF4 的显式 done 声明;②cgroup populated=0 = GF4 需要的物理背书。GF4 的 `ah job done` 显式协议对接①,LF1 cgroup 委托对接②。

### 复用现成入站通道(不新造)
- 信号入站复用 `master.ack_ready` 式 push-ingest 样板(research §五):hook/wrapper → `ah agent notify --event <e>` → RPC ingress → 写 PerceptionEvent。codex wrapper 发 `--event started` 即走此路。

## 七、执笔自述:两层的能力边界(REV1 更正,诚实不偷渡)

- **REV1 更正原版一处过头表述**:原版说"所有确认都等被动信号"——不准确。准确的两层划分是:
  - **状态层(生命周期状态)**:确实只用被动信号(hook 权威 + cgroup/pidfd),**禁 pane 造状态、禁注入探测**。
  - **动作层(我发对了没)**:用**核验自己写入动作的回显**(读 capture-pane 与 ground truth 等值比对),**这不是被动等待、也不是主动探测**——是主动核验一个刚做的写动作有没有生效。
- **仍然禁的**:注入探测字符主动问状态(探测层永久禁)。动作层核验**不注入探测字符**,与探测层不同性质。
- 能力边界:**动作层核验能确认"我发对且发出了",但确认不了"模型真开工了"**(那要 hook)。对无开始 hook 且无 wrapper 的 provider,开工侧只能等 cgroup populated→1 弱存活或超时上抛,**不退回注入探测**(codex 用 wrapper 造被动 started;agy 待验-2 未过同样退 wrapper)。
- 三处 待验(research §六)决定开工信号强度:待验-1(claude UPS)、待验-2(agy PreInvocation)、待验-3(codex Stop,不影响开工侧)。**凡依赖 待验 的信号,矩阵(§五)已标"依赖 X 否则走兜底",不假设 待验 必过。** 动作层核验不依赖任何 待验——它读的是自己写入的回显,各 provider 通用。
- **最弱区**:D Completion-Gate 的物理锁②依赖 cgroup 委托——WSL2 主目标已验证可用(`wsl2-cgroup-poc-result`),非委托宿主须降级(响亮、置信降)。
