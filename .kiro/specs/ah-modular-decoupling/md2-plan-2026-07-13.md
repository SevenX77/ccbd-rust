# MD2 解耦轮 · Wave 1 计划(草案,待 operator 审)

来源:严格按 `research/architecture-index.md`(v1,r1 ACCEPT + operator 复核通过)capability→owner 表选目标,不 ad hoc。目标筛选逻辑:index 里 1000+ 行的 ownership center 中,优先选**已有明确耦合违规证据或已有清晰内部子边界**的,而不是单纯按行数排序。

## 排序原则

不一次性铺开全部大文件。Wave 1 选 3 个,按**风险递增**排序——先用低风险目标验证"worktree/PR/边界文档/收口点全量 test"这套 MD2 流程本身能不能跑通,再碰高风险的核心模块。

---

## 目标 1(pilot,风险最低):`agent_io` 被动化

- **索引出处**:Layer 3 `agent_io`(`src/agent_io/`,4 文件共 ~1100 行)。
- **耦合证据**(已有实锤,非推测):`agent_io::reader` 直接引用 `db`(写状态)和 `marker`(Prompt 匹配),导致 Layer 3(物理宿主层)反向依赖 Layer 4(持久化)和 Layer 5(感知)。这条违规 o1 seed draft 就已指出,MD1 索引复核确认符号/路径属实。
- **边界假设**:`agent_io` 解耦后只做一件事——把 FIFO/PTY 轮询出的裸字节推到一个 `tokio::sync::broadcast` 或 mpsc 通道,不再感知 Prompt KB、不再直接改 DB。感知逻辑(vt100 解析 + MarkerMatcher 匹配 + db 状态转移)上移到 Layer 5,新起一个 `PerceptionStreamProcessor`(或并入现有 `marker`/`prompt_handler` 编排层,具体落点由实施者在 PR 里给出并让 r1 核)。
- **为什么排第一**:体量小(~1100 行,4 文件)、违规边界单一清晰(一个模块少两个不该有的依赖)、不触碰状态机/master 生命周期这类高风险路径,适合拿来验证整套 MD2 机制(worktree 隔离、PR 独立收口、r1 审、cargo 全量收口点)成本可控、能不能跑通。

## 目标 2(风险中等):`rpc::handlers::sessions.rs` 按域拆分

- **索引出处**:Layer 1 `rpc`(`src/rpc/handlers/sessions.rs`,2459 行)。
- **耦合证据**:该文件把三个不同域的 handler 混在一个文件里——(a)普通会话生命周期(`handle_session_create/kill/list`),(b)master pane 生成(`handle_session_spawn_master_pane` + 私有 `spawn_master_pane_inner`),(c)master cutover 全流程(`handle_session_master_cutover/handle_master_ack_ready/handle_master_tell_begin/handle_master_tell_failed`)。(c) 这部分的**领域逻辑已经在别处独立成模块**(`src/master_cutover.rs` 共享 helper + `src/db/master_cutovers.rs` 持久化),唯独 RPC handler 层没有跟着拆,是三个已分离领域被 handler 文件强行拉平。
- **边界假设**:拆成 `rpc/handlers/sessions.rs`(纯会话 CRUD/list)+ `rpc/handlers/master_cutover.rs`(master cutover 四个 handler,直接对应已存在的 `master_cutover`/`db::master_cutovers` 领域边界)。`handle_session_spawn_master_pane` 归哪边由实施者论证(它既是"会话内生成 pane"又是"master 相关"),PR 里必须显式给出理由。
- **为什么排第二**:纯代码搬移 + 调用点重连,不改变 RPC 方法名/协议契约(对外行为不变),但触碰的文件本身很大、调用面广(client/master_watch 都间接依赖这些 RPC 路径),风险高于目标 1、低于目标 3。

## 目标 3(风险最高,暂缓到 Wave 1 收尾且需要更仔细的边界评审):`monitor::master_watch.rs` 拆分

- **索引出处**:Layer 2 `monitor`(`src/monitor/master_watch.rs`,**5704 行,全仓库单文件最大**,是 Layer 3 那批总和的好几倍)。
- **耦合/内聚证据**:实测该文件只有 8 个顶层 `pub` 符号(`handle_master_death_detected`、`rearm_active_master_watches_on_startup`、`master_process_is_alive`、`master_watch_patrol_loop`、`resolve_master_watch_patrol_interval`、`spawn_master_pidfd_watch_task`、`monitor_key`、`MasterDeathSource`),却塞了 5704 行——典型"上帝函数"症状:几个函数体巨大,把探测、决策、级联清理、重试节流全部内联,而不是委派给已存在的 `master_revival`(928 行,决策助手)模块。
- **边界假设(粗,需要更细的设计评审)**:`master_watch` 应该只做"探测 + 调用 `master_revival` 做决策 + 调用 `monitor`/`platform` 做物理动作",而不是自己实现级联/重试逻辑。具体切法需要先过一轮设计(建议 d1 出边界稿,而不是直接派 codex 改),因为这是 master 自愈的核心安全路径,错了会复现 master 死循环/双活这类历史事故类型。
- **为什么排最后**:体量最大、风险最高(master 生命周期是全系统单点)、内部边界目前不够清晰(只有实施者读完全部 5704 行才敢下刀)。建议**先看目标 1/2 跑完的实证结果**(worktree 隔离是否顺利、CI 是否绿、r1 审有没有发现流程漏洞),再决定目标 3 是直接派 codex 还是先加一轮 d1 设计稿。

## Wave 2 候选(不在本轮批准范围内,先记账)

- `provider::home_layout.rs`(3039 行):`prepare_home_layout_*` 有 7 个变体(`_with_role`/`_with_extensions`/`_for_slot`/`_and_claude_credentials` 组合爆炸),像是参数累加出来的,而不是分层设计——需要先决定收敛成 builder 模式还是拆子函数,属于设计问题不是纯搬移,建议先过 d1。
- `db::system.rs`(3485 行):启动 reconciliation + system dump + 级联清理 + 过期 socket 清扫,四个不同职责挤在一个"system"文件里,是最大的 db 子模块也是最杂的一个。
- `db::state_machine.rs`(2978L)+ `db::jobs.rs`(2662L):o1 seed draft 提议的 Repository trait 抽象(`AgentStore`/`JobStore`)如果要做,这两个是核心,但改动面最大、影响所有上层调用方,应该放最后,且必须先过设计评审,不能直接派 codex 动手。

## 执行约束(沿用 MD2 需求原文)

- 每个 Wave 1 目标独立 worktree/PR,各自收口合入,不混批。
- 每个 PR 收口点本地跑一次全量 `CARGO_BUILD_JOBS=1 cargo test --workspace --test-threads=1`(真跑不是 check),CI 仍是最终门。
- 不保后兼容,可推倒重写内部实现;但目标 2/3 的 **RPC 协议契约对外不变**(client 不用跟着改)。
- PR 交接三元组显式记录(谁开的/谁盯 CI=master/谁验收=r1)。
- 三个目标各自的设计交付(哪怕只是 PR description 里的边界声明)必须先声明读了 MD1 索引哪些条目、改了哪些,过 MD3 门——这也是给 r1 审"是否发明了索引里已有的能力"提供依据。
