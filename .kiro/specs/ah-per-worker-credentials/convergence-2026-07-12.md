# Per-Worker Credentials · 评审收敛(master,2026-07-12)

> 输入:`design-divergence-2026-07-12.md`(o1,方案 A/B/C/D + 既有方案红队)。本文件是 master 对 o1 四套方案的二轮红队 + 收敛裁决,交 d1 执笔冻结前的"我带的事实"。d1 可挑战本收敛,不是终稿。

## 一、o1 未点透的一条关键分野(第三真相)

o1 方案 A 的时序图暗示"每次 401 触发的刷新都会重新走 keystore lookup",据此认为劫持 keystore 后端 = 拦到了刷新动作本身。**这一步推理有洞**:

- `keytar`/`libsecret` 是**纯凭据存取抽象**(get/set/delete 一个 secret blob),不是 HTTP 客户端。claude 进程真正发出的 OAuth 刷新 POST 是它自己的 HTTP 逻辑,不经过 keytar。
- 结合已坐实事实④"RT 缓存在内存、刷新前不重读文件"——这条缓存行为发生在**凭据对象已从任意后端(文件或 keystore)解析出来之后**的那一层,与具体后端无关。也就是说:如果 claude 在进程启动时 lookup 一次、之后全进程生命周期只认内存,那么**换成 keystore 后端劫持,同样换不掉"进程内存里那份 RT 不会被外部改写"这个根问题**——方案 A 并不比方案 B 更能拦住"worker 独立发起会轮换的刷新"这个网络请求本身;它只是把"进程启动时拿到的初始凭据"从文件通道换成了 keystore 通道。
- **结论**:方案 A 的核心价值被 o1 高估了。它不是"更底层的仲裁点",而是"和方案 B 同一层(进程启动时的凭据注入点),只是换了个更复杂、更脆的信道"。除非 spike 证实 claude 在每次拟刷新前都会重新 lookup keystore(而不是只信内存),方案 A 相对方案 B 没有额外的不变量保证,却多背负 LD_PRELOAD/D-Bus 仿冒的工程脆弱性(沙箱擦除环境变量、CLI 升级后静态链接击穿——o1 自己也红队出这两条)。
- **真正在"协议边界"上、与 claude 内存缓存行为无关的仲裁点只有一个:方案 C(拦截 `platform.claude.com/oauth/token` 这一个 HTTP 端点)。** 不管 claude 内部何时决定要刷新、RT 缓存在哪、进程活了多久——只要它想刷新,请求必须打这个网络端点,proxy 在这里 single-flight 是**不依赖任何 claude 内部实现细节的硬保证**,而方案 A/B/D 全部依赖"claude 不会在到期前提前用内存里的旧 RT 尝试刷新"这类行为假设。

这条分野决定了收敛方向:**方案 C 不是"网关秒死教训后的降级备选",而是唯一能把核心不变量做成"协议级硬保证"而非"行为假设"的方案**,应当在架构里占据"安全网"角色,即使它单独跑起来复杂度不是最低。

## 二、对四套方案的收敛裁决

| 方案 | 收敛定位 | 理由 |
|---|---|---|
| **方案 A(keystore/so 劫持)** | **降级为可选 spike,不进入本轮实现范围** | 红队后价值被高估(见一);工程脆弱性全项目最高;不满足"改一次 CLI 就全崩"的可维护性底线。留一条平价 spike(验证 claude 是否 per-refresh 重新 lookup),证实了再考虑替换方案 B 的信道。 |
| **方案 B(去 RT 明文拷贝 + 保鲜)** | **采纳为主路径(Layer 1,主动层)** | 复杂度最低、无需代理/劫持;红队死穴("长任务内存不重载")用方案 D 的空闲滚动重启兜底,而不是否决方案 B 本身。 |
| **方案 C(oauth/token 定点代理)** | **采纳为安全网(Layer 2,协议级硬保证)** | 见一:唯一不依赖 claude 内部缓存行为假设的方案。作用域收窄到只拦一个域名+一个路径,不是全局网关(区别于 #146/#147/#149 秒死教训——不代理任何模型请求流量,爆炸半径大幅收窄)。前置门槛是 SSL pinning + HTTPS_PROXY 遵从,需 spike 先证。 |
| **方案 D(RT 剔除 + 到期欺骗 + 空闲滚动重启)** | **采纳为 P2 恢复机制(Layer 3,失败隔离/可观测层)**,不单独作为主防线 | 单独作为主防线的死穴是"超长任务无法无损重启";但作为 Layer 1(方案 B)的配套恢复动作——worker 在空闲态被重启以加载新鲜凭据、且给"某 worker 卡在无法刷新的状态"提供一个可观测+可恢复的操作口径——完全贴合 P2("失败可观测、可独立恢复")的要求。 |

## 三、收敛后的架构(交 d1 执笔冻结的骨架)

**三层防御,不是单选一套方案**:

1. **Layer 1(主动,日常路径)**:`ahd` 持有唯一真实、含真 RT 的凭据(来自用户那次真实登录,**绝不下发到 worker 沙箱**),运行单一后台刷新 loop,在 access token 剩余 TTL ~20% 时主动刷新;每次刷新后,原子覆盖写入各 worker 沙箱内的**阉割版**凭据文件(仅 `accessToken`+`expiresAt`,`refreshToken` 字段整体去除/填 dummy 值)。worker 永远不掌握真 RT ⇒ 即使 worker 自己尝试刷新,请求在到达上游前就因 RT 非法而失败,不产生服务器端令牌轮换。
2. **Layer 2(安全网,协议级硬保证)**:本地 HTTPS 定点代理**只**拦截 `platform.claude.com` 上的 `/v1/oauth/token` 路径(模型 API 请求 100% 直连,不代理,规避网关秒死教训);任何 worker(不论何种原因)真的发出了刷新请求,在此处 single-flight 收拢为一次真实上游调用,回填给所有等待方。这一层不依赖 Layer 1 是否生效、不依赖 claude 内部缓存/重读行为假设,是核心不变量的兜底证明。
3. **Layer 3(失败隔离与恢复,对应 P2)**:worker 凭据判定失败(无法刷新、无法工作)时,作为**可观测事件**上抛(日志行 / `ah ps` sub_state / 事件流,具体接哪个由 d1 在冻结稿里定,需先审计现有失败路径,见 P2 testability 备注);`ahd` 在该 worker 处于 IDLE 时对其做平滑重启以重新加载新鲜凭据映像,给出独立于其他 worker 的恢复路径。

**WSL2 defect B 天然消除**:Layer 1 的 worker 凭据文件是 `ahd` 主动物化的独立文件,不是指向 `/mnt/c` 的 symlink,不存在跨设备 rename 退化 copyFile 写穿宿主的路径——验收硬项("任何路径不得写穿 `/mnt/c`")在此架构下是结构性满足,不需要额外防御代码。

## 四、优先 spike 顺序(交 d1/实施线前必须先出结论的 load-bearing 假设)

按"最便宜 + 最能一票否决/一票转向"排序,不是 o1 建议的顺序:

1. **Layer 2 生死门槛,最先做**:`platform.claude.com` 的 `/v1/oauth/token` 是否有 SSL pinning、是否遵从 `HTTPS_PROXY`?——`mitmproxy` 起本地代理 + 设置 `HTTPS_PROXY`,跑一次真实 claude 刷新观察。**如果这一步失败(pinning 挡死),Layer 2 整层要换成别的兜底机制(比如退回纯 Layer 1 + Layer 3,不变量退化为"极高概率不发生"而非"协议级保证"),这是唯一可能推翻整个收敛架构骨架的假设,必须第一个出结论**。
2. **Layer 1 生死门槛**:claude 在凭据文件里**没有** `refreshToken` 字段(或为 dummy)时,启动/使用期间遇到 401,是否老实进入"需要重新登录"的失败态,而不是 crash 或死循环重试?——手动改本机 `~/.claude/.credentials.json` 的 `refreshToken` 观察行为。
3. **Layer 3 前置**:审计 claude 当前遇到凭据失效时,现有代码里失败路径落在哪(日志/退出码/hang),再决定 P2 的"可观测"具体接口。
4. **方案 A 的 keystore-per-refresh-lookup 假设**:降为可选、非 blocking,仅在 1-3 都通过且时间允许时再做,证实了才评估是否值得把 Layer 1 的信道从文件换成 keystore(收益边际,不预期会推翻架构)。

## 五、待 d1 执笔冻结时明确的开放点(非 blocking,design-rev.md 需写清但不改变架构方向)

- Layer 1 保鲜阈值(剩余 TTL 20% 触发刷新)是否需要可配置,还是写死。
- Layer 2 代理的具体实现载体(ahd 内嵌 tiny TLS 拦截 vs 独立子进程)、CA 证书注入到 worker 沙箱的机制。
- Layer 3 "空闲"判定的具体信号源(现有 job/agent 状态机里哪个字段)。
- 验收判据钉死为 **A2 三层验收第三层**:真二进制 spawn 真 claude worker,在真实(或受控可复现的)401/RTR 场景下端到端跑通,不允许只用 `--lib` 假上游测试放行本条不变量的最终验收(单测仍然需要,但不是唯一证据)。
