## What This Draft Reuses vs. What It Adds

**Reused, not re-answered — the hook attribution race.** `ah-perception-arbiter/design.md` Must-Answer Q4 already pinned the attribution mechanism: durable local write before any socket attempt, daemon-side inotify + cold-scan-on-restart consumption, and attribution by a **per-dispatch-attempt cookie** (`AH_JOB_ATTEMPT_COOKIE = "{job_id}:{dispatch_seq}"` or equivalent — exact format pinned by whoever implements it, arbiter Q4 correction 2026-07-10) that does **not** depend on the sender process still being alive. Track A treats that decision as settled input. R1 below adds only the **delivery transactionality** the arbiter left as a one-line promise ("at-least-once delivery across daemon restart") — the ACK/reap lifecycle, the event-id idempotency ledger, the replay ordering, and the error-book — without re-opening who-owns-which-event or how attribution keys are minted. Where R1 and arbiter Q4 describe the same physical artifact (the outbox directory), R1 is the transport spec and arbiter Q4 is one consumer (`perception_events` insert) of what the transport delivers.

**Red lines observed** (`perception-final-convergence-2026-07-09.md` §三, two refuted claims): this draft does **not** invoke any K8s probe-absence optimistic/pessimistic asymmetric default, and does **not** invoke any K8s-API-layer-forced-`/status` single-write argument. R1's replay leans only on the *confirmed* level-based re-derivation over re-readable durable artifacts (final-convergence §1.5) — the outbox file is exactly such a re-readable artifact; a transient socket call is not, which is why durability must precede delivery.

## Design Thesis

Today `ah agent notify` is a synchronous RPC and nothing else. `src/bin/ah.rs:532` calls `client.call("agent.notify", params)`; on any error — ahd not running, socket stale, daemon mid-restart — the hook's `ah agent notify` process exits non-zero and **the event is gone** (`src/bin/ah.rs:547`). The only durable trace is `append_hook_debug_log` (`src/bin/ah.rs:578`), which is a human-debug appender, never replayed, never consumed. This is precisely the G1 structural defect the north-star names ("notify 无 ACK 无记账,ahd 不在=事件蒸发", first-principles §三 G1 row): the agent→ahd segment of the T1 event spine is fire-and-forget, so it is not a spine at all — it is a wire that drops packets whenever the far end blinks.

R1 makes that segment transactional. The hook stops *sending* an event and starts *journaling* one: it writes a durable outbox record first, then attempts fast-path delivery as an optimization, and exits safe regardless of whether delivery landed. ahd consumes the outbox at-least-once and reaps only after a durable commit; on restart it cold-scans the outbox before serving traffic, so a `kill -9` of ahd mid-flight loses nothing. This is the standard transactional-outbox pattern, and it is the load-bearing floor for everything above it — R2's explicit-done declaration, F2 turn boundaries, F4 waiting-for-input — all ride this same transport (first-principles §四 R1: "其余一切的地基").

G4 is the prophylaxis for the disease family the north-star calls "零件好的忘了装" (first-principles §三 G4 row): the transport can be perfectly built and still deliver nothing if the hook was never wired into the agent's config, or its config was edited/deleted out from under ahd, or a probe silently stopped reinstalling. R1 guarantees *delivery of events that fire*; G4 guarantees *events actually fire* — that the hook config on disk still matches what ahd expects, that a synthetic event round-trips through the real path, and that a missing or drifted wire raises a loud alarm and self-heals instead of failing silent. Without G4, R1's durability is a guarantee about an empty pipe.

```text
  R1 (this draft):  [hook fires] --journal--> [outbox] --at-least-once--> [ahd consume] --reap-after-commit-->
  G4 (this draft):  assert the hook is wired, its config matches, and a synthetic event survives the whole path
```

Neither R1 nor G4 decides `agents.state` or `jobs.status`. R1 is a transport; its delivered events feed the arbiter (perception verdicts) and Track B's completion protocol (job done-declaration). G4 is a self-check; its findings feed logs, `ah doctor`, and startup gating. The write authority over state stays where the arbiter design put it.

# Part R1 — Hook Delivery Reliability (Outbox / ACK / Replay)

## R1-Q1: Where does durability begin — and what exactly is written before any socket attempt?

**Decision: durability begins at an atomic `rename()` into a per-agent, host-visible outbox directory, and it happens before the hook makes any RPC call.** Concretely, the rewritten `ah agent notify` flow is:

1. Construct the event record in memory: `{ event_id, agent_id, provider, event, attempt_cookie, hook_fired_at, payload }`. `event_id` is a content-independent unique id minted by the hook (ULID/UUIDv7 — monotonic-ish for scan ordering; the hook already has a `--event-id` surface, `src/bin/ah.rs:518`). `attempt_cookie` is the arbiter-Q4 per-dispatch-attempt cookie read from the environment; R1 does not mint it.
2. Write that record to `{outbox_dir}/{event_id}.tmp`, `fsync` the file, then `rename()` to `{outbox_dir}/{event_id}.json`, then `fsync` the containing directory. The rename is the durability commit point — POSIX guarantees the reader (ahd) never observes a partial file under that name. *(Naming pinned to `{event_id}.tmp → {event_id}.json` to match arbiter Q4 and Track B — F-7; the host-side scanner globs `*.json` and must never see `.tmp`.)*
3. **Only then** attempt the fast-path: `client.call("agent.notify", …)` exactly as today, but treat its result as an *optimization*, not the delivery guarantee. Success lets ahd process the event immediately and reap the file; failure (ahd down, socket stale) is a non-error — the durable file is the guarantee, and the hook exits `0`.

**The durability commit can itself fail — and that must be loud, not exit-0 (F-1).** The exit-0-safe posture above is scoped **only** to step-3 RPC failure. If step 2 — the durability commit — fails (`ENOSPC` mid-write, `fsync` `EIO`, `rename` `EROFS`, a full or read-only outbox, a missing/undwritable `outbox_dir`), then **nothing durable landed** and the hook must **exit non-zero with a provider-visible error on stderr**, never exit 0. Exiting 0 on a failed journal is the exact "发射即忘进虚空" (fire-and-forget into the void) disease R1 exists to kill, re-entering through the back door: the hook would signal success while the event evaporated. Concretely: the `.tmp` write, its `fsync`, the `rename`, and the directory `fsync` are each checked; any error ⇒ `tracing::error!` + best-effort cleanup of a partial `.tmp` + non-zero exit. This is distinct from — and the mirror image of — today's bug: today the hook exits **non-zero on RPC failure** (`src/bin/ah.rs:547`), which R1 correctly flips to exit-0-safe; R1 must **not** simultaneously let a *journal* failure become exit-0. The invariant is: **exit 0 ⇔ a durable outbox record exists.** (The outbox-inaccessible case remains a loud escalation to the sandbox design per arbiter Q4, not a route-around; F-1 adds the *transient* write-failure branch the arbiter's access-denied clause did not cover.)

`outbox_dir` is the same host-side directory arbiter Q4 already introduced (`{agent_home}/outbox/`). It must be a path the sandboxed hook can write and ahd can read; if the sandbox model cannot guarantee outbox write access, that is a **loud escalation to the sandbox design, not a route-around** (arbiter Q4 failure-mode, carried forward verbatim — do not silently drop).

**Rationale.** The current order is exactly inverted: today the socket call *is* the delivery, so far-end unavailability equals data loss. Reversing to journal-first is the whole content of the G1 fix. `rename()` over the same filesystem is atomic and is the only primitive in this path that survives a `kill -9` of both ends. The fast-path RPC is kept purely to preserve today's sub-second common-case latency — without it every event would wait for ahd's next inotify/scan tick — but it is demoted from *the* mechanism to *an* accelerator.

- **North-star acceptance:** §五.1 ("任一时刻 kill -9 ahd 再拉起,事件流无洞") — the rename commit point is what makes "no holes" provable.
- **Dogfood node:** new ledger row **R1-outbox-replay** — dispatch a job, `kill -9` ahd while the agent's Stop hook is firing, relaunch ahd, assert the completion event is consumed exactly once (see R1-Q3 harness). Also retires the "hook 信号偶发丢失" incident family (handoff §域1) whose root cause arbiter Q4 already named as the sd_notify send-before-PID1-processes race.

## R1-Q2: At-least-once means duplicates — what makes redelivery safe?

**Decision: a dedicated transport-level dedup ledger, checked at the outbox-consume boundary *before* the record is routed by `kind`; state-CAS is demoted from the idempotency mechanism to a secondary guard.** Every outbox record carries a stable `event_id` (R1-Q1). The consume loop's first step, for *every* record regardless of `kind`, is: `INSERT INTO {dedup_table}(event_id) … ON CONFLICT DO NOTHING`; zero rows affected ⇒ already applied ⇒ drop and reap without dispatching. Only a first-seen `event_id` proceeds to be routed to its handler. The dedup insert and the handler's effect commit in **one transaction** so a crash between apply and reap cannot double-apply.

**Why the ledger must live at the transport boundary, not on the `events` table (F-2 — corrected against schema).** My earlier "reuse the `events` table's `event_id` + `UNIQUE(event_id)`" was structurally wrong, verified against `src/db/schema.rs`:
- The `events` table has **no `event_id` column** (`schema.rs:124-131` — columns are `seq_id, agent_id, request_id, event_type, payload, created_at`); `event_id` is buried inside the JSON `payload` string (`state_machine.rs:727`). You cannot put `UNIQUE(event_id)` on it without a migration (generated column / expression index).
- The one existing idempotency index is `UNIQUE(agent_id, request_id) WHERE request_id IS NOT NULL` (`schema.rs:135`) — keyed on `request_id`, and the hook-event INSERT passes `request_id = NULL` (`state_machine.rs:733`), so it doesn't even cover today's hook path. (This *confirms* the "CAS is the only current guard" motivation below — but it means there is nothing to "reuse.")
- **Decisively for the cross-track seam:** Track B's `job_done`/`job_fail` records are consumed onto a **different table**, `job_transitions` (`apply_job_done_declaration_sync`, Track B MA-1), whose PK is `job_event_id INTEGER PRIMARY KEY AUTOINCREMENT` (`schema.rs:176`) — a DB-minted integer in a *disjoint namespace* from the hook-minted ULID `event_id`, with its own `kind CHECK` (`schema.rs:180`). A `UNIQUE(event_id)` bolted onto `events` would give Track B's declarations **zero** dedup, so under at-least-once redelivery a replayed `job_done` file would be **double-applied** to `job_transitions`. The transport (Track A) owns delivery, so the dedup that makes redelivery safe must be **one ledger over all outbox records, keyed on the outbox `event_id`, applied before the `kind` fork** — covering both the F2 `events` path and the F3 `job_transitions` path with a single guard. This is a real schema delta (a new dedup table, e.g. `outbox_consumed(event_id TEXT PRIMARY KEY, consumed_at INTEGER)`), **not** free reuse (F-5).

**This is a pinned Track-A/Track-B contract, not a unilateral choice.** Track B's MA-1 must route `job_done`/`job_fail` through this same consume boundary (dedup-then-route), and must **not** assume the `events`-table `event_id` covers it. Both drafts previously assumed different homes for the same idempotency key; this section fixes the home to the transport ledger and Track B's merge must adopt it.

**Why CAS alone is insufficient — corrected against current code.** Today idempotency rides entirely on the state-version CAS inside `mark_agent_idle_hook_event_outcome_sync` (`src/db/state_machine.rs:619-632`): a redelivered event finds the agent no longer `BUSY`/`WAITING_FOR_ACK` and is swallowed (`:623`). That is accidental idempotency, and it breaks under exactly the sequence at-least-once makes common: event E delivered once (BUSY→IDLE), a new job re-arms the agent to BUSY, then E is redelivered (outbox replay after an ahd restart that hadn't yet reaped E's file) — CAS now sees BUSY and **misapplies E to the new job**. An explicit `event_id` ledger closes this: E is recognized as already-seen regardless of current state. CAS stays as defense-in-depth (it correctly rejects genuinely-stale transitions), but it is no longer the thing standing between replay and corruption.

**Rationale.** at-least-once + idempotent-consumer is the only combination that is both crash-safe and duplicate-safe; exactly-once delivery over a crashing transport is unachievable, so we make duplicates harmless instead of trying to prevent them. Keying on a producer-minted `event_id` (not on content or on "current active job") is what lets a duplicate be recognized after the world has moved on.

- **North-star acceptance:** §五.1 (replay must not corrupt) and §五.5 (nailed by an automated test: the redeliver-after-rearm sequence above is a required RED test).
- **Dogfood node:** **R1-outbox-replay** (same harness, duplicate-delivery arm) — assert a doubly-scanned outbox file transitions state exactly once.

## R1-Q3: How does ahd replay after restart, and what happens to events it can never apply?

**Decision: cold-scan the outbox before serving, replay in `event_id` order, and quarantine the unapplyable into an error-book rather than dropping or hot-looping on them.** On startup, before ahd accepts new RPC traffic, it enumerates `{agent_home}/outbox/*.json` across all agent homes and feeds each through the same idempotent consume path as a live inotify event (R1-Q2). This is the "ahd 回来先读错误本重放" clause of the G1 fix (first-principles §三 G1). Steady-state consumption is inotify-driven; the cold-scan is the restart-recovery and the inotify-miss backstop (a periodic sweep also catches any inotify drop, since the durable file is level-triggered and re-readable — final-convergence §1.5).

**Ordering.** Replay oldest-first by `event_id` (ULID/UUIDv7 embeds a timestamp prefix, giving scan order ≈ fire order). Strict global ordering is *not* required for correctness because each event's effect is idempotent and attributed by cookie, not by arrival position — but approximate fire-order replay avoids needless CAS churn (applying a stale-then-fresh pair in order settles in one transition instead of two). *Reserved-prefix ids are exempt from this assumption (F-6):* the G4-Q4 selfcheck record uses a fixed non-ULID id (`selfcheck:{agent_id}:{boot_id}`) that does not time-sort — it is routed to a no-op sink (G4-Q4), so its replay position is irrelevant, but it still takes a dedup-ledger row (R1-Q2) so a crash-surviving selfcheck file re-scans as a harmless no-op rather than re-running.

**Error-book (the quarantine, not a drop).** An outbox file that (a) fails to parse, (b) references an `attempt_cookie` that cannot be attributed to any known dispatch, or (c) has failed consume N times (N=3, matching the arbiter/evidence-gate "third-attempt escalation" convention) is `rename()`d into `{agent_home}/outbox/dead/` with a **loud** `tracing::error!` carrying the file name and failure reason. It is never silently deleted and never left to hot-loop the scanner. The dead-letter directory is an operational surface (a `DoctorCheck` in G4-Q5 counts it) and a dogfood data point, not an auto-retry queue.

**Rationale.** Cold-scan-before-serve is the arbiter's own adopted mechanism (arbiter Q4: "inotify + cold-scan-on-restart"); R1 pins its *timing* (before serving, so no window exists where ahd is up but hasn't replayed) and its *failure handling* (error-book, so a single poison file can neither vanish nor wedge the scanner). Quarantine-with-loud-log over silent-drop is the north-star's "响亮地降级" applied to the transport's own edge cases (first-principles §二 closing principle).

- **North-star acceptance:** §五.1 (the canonical `kill -9 ahd` → relaunch → "事件流无洞,outbox 重放可证" scenario — this section *is* that clause's mechanism).
- **Dogfood node:** **R1-outbox-replay** (restart arm) + new row **R1-deadletter** — inject a malformed outbox file, assert it lands in `outbox/dead/` with a loud log and does not stall replay of its siblings.

## R1-Q4: What is the ACK, and when is an outbox record safe to delete?

**Decision: the ACK is ahd's durable DB commit of the event's effect + ledger row; reaping (file deletion) happens strictly after that commit, and the hook process never blocks on it.** Two distinct acknowledgements exist and must not be conflated:

- **Producer-side "ACK":** there is none, by design. The hook's guarantee is the `rename()` (R1-Q1); it exits `0` immediately after, whether or not ahd was reachable. Making the hook wait for a daemon ACK would reintroduce the coupling R1 exists to remove (and would stall the agent's turn on ahd latency).
- **Consumer-side ACK = commit-then-reap.** ahd applies the event and inserts the `event_id` ledger row in one transaction (R1-Q2); the outbox file is deleted only after that transaction commits. If ahd crashes between commit and reap, the file survives, the next cold-scan re-reads it, and the ledger makes the re-read a no-op (R1-Q2). If ahd crashes between read and commit, the file also survives and the event is simply applied on restart. Either crash window is safe; there is no ordering of crashes that loses or double-applies an event.

**Rationale.** This is the transactional-outbox invariant stated precisely: *the file may only be destroyed by the party that has durably recorded its effect.* The north-star's G1 phrasing "先journal后投递等ACK" is satisfied not by the hook waiting for an ACK (it must not) but by the *reap* waiting for the *commit* — the acknowledgement that matters is ahd's durable record, and the file is its receipt until then.

**Consistency with the arbiter's 2s hook budget (arbiter Q2).** The arbiter's 2s hook-grace window measures "how long ahd waits to *notice and process* an already-durably-written outbox file" (inotify latency + tick cadence) — it does **not** measure disk I/O or hook liveness, and it does **not** produce a direct job-terminal write (arbiter Q2 correction: the expiry yields a perception verdict feeding Phase-2 evidence-check, never a one-way guillotine into `Failed`). R1 is fully consistent with that: the outbox makes the write durable in step R1-Q1, so the 2s clock only ever runs against ahd's own consume latency. If that window ever needs a different value, arbiter Q2 already named the correct conclusion — "the outbox consumption loop's cadence assumption was wrong," not that R1 needs its own second number.

- **North-star acceptance:** §五.1 (no-loss under crash) — the commit-then-reap ordering is what makes the crash-window analysis close.
- **Dogfood node:** **R1-outbox-replay** (crash-between-commit-and-reap arm — hardest to reproduce; harness uses a fault-injection hook that `abort()`s ahd after commit, before reap, and asserts exactly-once on relaunch).

# Part G4 — Control-Path Self-Check

## G4-Q1: What does "the control path is wired" actually assert — what are the check targets?

**Decision: three concrete, machine-checkable assertions, in ascending cost.** "Wired" is not a vibe; it decomposes into exactly these targets, and the three startup tiers (G4-Q2) each cover a prefix of them:

1. **Config presence & shape** (cheap, read-only): for every agent ahd believes it manages, the agent's provider config file on disk contains the ah-owned `Stop` hook entry (and `UserPromptSubmit` where the provider uses it), with a `command` string that (a) invokes `ah agent notify`, (b) carries the correct `--agent-id`, (c) points at the *current* ahd socket path, and (d) names the right `--provider`. **This target is per-provider-shape-aware (F-4):** it locates the ah-owned entry using the same `is_ah_owned_hook_item` predicate the injector uses (`command.contains("ah agent notify")`, `src/provider/home_layout.rs:1202-1207`), inside the correct file+nesting for each provider — see the shape table in G4-Q3. The `command` itself is a pure string comparison against what `build_ah_hook_command` (`:674`) renders today (shared across all three providers).
2. **Config fidelity** (cheap): the ah-owned hook entries on disk match what ahd would render — a **scoped** comparison over *only ah's own entries* (G4-Q3), **not** a whole-config fingerprint. (Whole-config hashing is what F-3 showed false-blocks on benign operator edits; it is deliberately not used here.)
3. **End-to-end liveness** (expensive, side-effecting): a synthetic event, injected at the real hook entry point of **each provider's installed hook**, actually traverses hook → outbox → ahd consume and is observed as consumed (G4-Q4).

**Rationale.** The G4 disease family is "a good part nobody installed" — the failures are all *absence* or *drift* of ah's own wiring, not logic bugs, so the checks are all *comparisons of ah's-own-entries-on-disk against ahd's own expectation*. ahd already holds the expectation: it rendered the hook command (`build_ah_hook_command`) and it knows which entries are its own (`is_ah_owned_hook_item`). G4 is *reusing what ahd already knows about its own hooks* to check what's actually on disk — scoped to ah's entries so an operator's out-of-band additions are invisible to the check (the F-3 correction), and enumerated across all three provider shapes so no provider is silently unmonitored (the F-4 correction).

- **North-star acceptance:** §五.4 ("手工删掉某 agent 的 hook 配置:下次启动自检报警并自动修复") — target (1) is what catches the manual-delete; targets (2)/(3) catch subtler drift.
- **Dogfood node:** new ledger row **G4-wiring-selfheal** (see G4-Q6 for the self-heal arm).

## G4-Q2: What runs at startup, and what blocks vs. warns — the three-tier check?

**Decision: light / medium / deep, with light always-on and blocking-with-repair, medium always-on and non-blocking, deep opt-in via `ah doctor`.** The tiers map to G4-Q1's targets by cost:

| Tier | When | Covers | On failure |
|---|---|---|---|
| **Light** | Every ahd startup, and every `ah up`/realign that (re)materializes an agent home | Target 1 (presence/shape) + Target 2 (scoped ah-owned-entry fidelity) — both read-only, sub-millisecond per agent, over **ah's own entries only** (F-3) | **Loud alarm + auto-repair** (re-materialize ah's own hook entries, G4-Q6) + emit a self-check event. Blocks the agent from being considered dispatch-ready until the repair is confirmed by a re-check. Does **not** block ahd's own boot (other agents proceed). Because the check is scoped to ah's own entries, a benign operator edit elsewhere in the config never trips it (the F-3 non-convergence is structurally impossible). |
| **Medium** | Every ahd startup, after light passes | Target 3 (synthetic round-trip, G4-Q4) for each agent | Loud alarm + self-check event; **non-blocking** — a synthetic-trigger failure warns and surfaces in `ah doctor`, but does not itself hold dispatch (light already guaranteed the config is present/correct; medium catches path-level rot the config can't reveal). Rationale for non-blocking below. |
| **Deep** | On demand: `ah doctor` (never in the hot startup path) | Full end-to-end incl. real daemon consume confirmation, dead-letter census (R1-Q3), cross-agent wiring matrix | Reported as `DoctorStatus::Fail`/`Warn` `DoctorCheck`s (`src/cli/doctor.rs:15`); operator-facing, no auto-action. |

**Why light blocks but medium does not.** Light checks a *fact ahd fully controls* (the config it wrote is still what's on disk) and can *fully remediate* (re-materialize) — so blocking-until-repaired is safe and correct: dispatching to an agent whose hook is missing would resurrect the exact fire-and-forget-into-the-void failure R1 fixed. Medium exercises the *live daemon path*, which can fail for reasons ahd cannot instantly fix (a transient socket race during its own boot, an inotify not-yet-armed); making that block startup would risk a boot deadlock (ahd refusing to serve because its own not-yet-serving socket didn't answer a synthetic probe). So medium warns loudly and defers the hard verdict to deep/`ah doctor`, where the daemon is unambiguously up.

**Rationale.** The north-star specifies "启动时三档检查(轻/中/深);深度端到端检查挂 `ah doctor`" (first-principles §三 G4) — this table is that clause, with the block/warn boundary drawn at the line between "ahd can both detect and fix this itself" (block+repair) and "ahd can only report this" (warn/doctor). The point of three tiers is that the cheap always-on check must never be gated behind the expensive one — the manual-delete scenario (§五.4) has to be caught on *every* boot at near-zero cost, which only the light tier affords.

- **North-star acceptance:** §五.4 (light tier is the always-on catcher) + §五.5 (each tier's trigger is a required automated test).
- **Dogfood node:** **G4-wiring-selfheal** (light-tier arm: delete a hook config, restart ahd, assert alarm + repair + re-check pass before dispatch).

## G4-Q3: How is config drift detected — and does it catch a manual delete?

**Decision: drift detection compares *only ah's own materialized hook entries* on disk against what ahd would render — scoped by the `is_ah_owned_hook_item` predicate, and located per-provider-shape — NOT the whole-config `compute_config_hash`; a delete is the degenerate drift case (the ah-owned entry is absent). (F-3 + F-4.)** The check, per managed agent, per provider shape: re-read the provider's config file, find the ah-owned `Stop` hook item(s) (`command.contains("ah agent notify")`, `src/provider/home_layout.rs:1202-1207`), and compare that item's `{command, timeout}` against what `build_ah_hook_command`/`hook_timeout_for_provider` render now. Absent entry ⇒ delete (strongest alarm, names the missing entry). Present-but-different `command`/`timeout` ⇒ drift. Everything else in the file — operator hooks, skills, settings, plugins — is **out of scope** and never enters the comparison.

**Why not whole-config `compute_config_hash` (the F-3 correction).** Verified: `compute_config_hash` (`src/provider/fingerprint.rs:45-79`) hashes the *entire* `hooks` map + `settings` + `plugins` + `skills` + `bundle` (`:61-74`), not ah's entries in isolation. Combined with the light-tier "block dispatch until a re-check passes" and G4-Q6's "merge, don't clobber operator config," a whole-config fingerprint is **self-contradictory**: an operator makes any out-of-band edit (their own hook/skill/setting — a case G4-Q6 explicitly designs for) → whole-config hash ≠ ahd's expectation → drift fires every boot → merge-preserving repair keeps the operator's edit → on-disk *still* ≠ ahd's isolated expectation → re-check never passes → the agent is **permanently blocked from dispatch on a benign edit.** Scoping the comparison to ah's own entries dissolves this: the operator's additions are invisible to the check, so repair→re-check converges, and the block only ever holds for a genuinely missing/wrong ah entry — which repair *can* fix. (This does mean G4 no longer shares one definition of "correct config" with realign's whole-home `compute_config_hash`; that is intentional — G4 asks a *narrower* question, "is ah's own wire intact," which is the only question whose answer ahd can both detect and fully remediate.)

**Per-provider config-file shapes — enumerated, not "claude then same for others" (the F-4 correction).** Track B §5.0/§5.7 (code-verified) established all three providers wire the same observe-only `Stop` hook, each in a **different file with a different nesting**. G4 must check each shape explicitly — I re-verified each in `src/`:

| Provider | File | Nesting of the ah-owned Stop entry | Materialize / inject path | Extra gate |
|---|---|---|---|---|
| **claude** | `settings.json` | `hooks.Stop[]` (claude settings schema), pushed as `materialized_ah_hook(ctx,"Stop")` | `materialize_claude_settings` (`home_layout.rs:892`), Stop push `:232` | — |
| **antigravity / agy** | `hooks.json` | top-level wrapper key **`ah-completion-push`** → `Stop[]` → group `{matcher: "", hooks:[{command,…}]}` | `inject_antigravity_hook_push` (`:408-428`) / `merge_antigravity_hooks` (`:358`) | — |
| **codex** | `hooks.json` | top-level wrapper key **`hooks`** → `Stop[]` → group `{matcher: "*", hooks:[{command,…}]}` | `merge_codex_hook_push` (`:1167-1190`) | also requires `features.hooks = true` in codex config (`:1163`); a check that ignores this reports "wired" while codex silently never fires the hook |

The two `hooks.json` shapes are **not** interchangeable (agy wrapper `ah-completion-push` + matcher `""`; codex wrapper `hooks` + matcher `"*"`), so the presence/drift check needs a per-provider locator, not one path reused. The `command` string being compared is provider-agnostic (all route through `build_ah_hook_command`), but *where it lives* is not.

**The manual-delete scenario, concretely.** "手工删掉某 agent 的 hook 配置" manifests two ways, both caught, **for whichever provider**: (a) the ah-owned `Stop` entry is gone from that provider's file — the presence check (Target 1) fails and the alarm names the specific provider + missing entry; (b) the whole config file is gone/empty — "no ah hook wired at all," the strongest alarm. Light-tier repair re-materializes **ah's own entries** via the same path `ah up` uses for that provider (claude `materialize_claude_hooks`/`materialized_ah_hook` `:227-235`; agy `inject_antigravity_hook_push`; codex `merge_codex_hook_push` — each already merge-preserving via `remove_ah_owned_hook_groups` then re-push, `:419/:1180`) and re-checks; only a passing re-check clears the dispatch block.

**Rationale.** The presence pre-check exists because a bare mismatch alarm is useless to an operator; naming the specific provider + missing entry is what makes the loud log actionable ("响亮地降级" made specific — say *what* is unwired and *for which provider*, not merely *that* something is). Scoping to ah-owned entries (F-3) is what makes the block+repair loop *convergent* under benign edits; enumerating the three shapes (F-4) is what makes the check *complete* — a deleted codex or agy hook is exactly the "零件好的忘了装" this self-check exists to catch, and a per-provider silent completion loss is indistinguishable from 假完成 (Track B §5.7).

- **North-star acceptance:** §五.4 (this is the detection half of that clause; G4-Q6 is the repair half).
- **Dogfood node:** **G4-wiring-selfheal** (drift arm: hand-edit an ah-owned hook `command` to point at a stale socket, restart, assert ah-entry-mismatch alarm + repair) — run the arm **once per provider shape** (claude `settings.json`, agy `hooks.json`, codex `hooks.json`), since the locator differs; plus a benign-edit arm (operator adds a non-ah hook/skill → assert **no** alarm and dispatch stays unblocked, proving the F-3 scoping).

## G4-Q4: What is the synthetic-trigger check — how do we know the wire actually carries current?

**Decision: inject a self-check event through the real `ah agent notify` entry point with a reserved self-check `event_id` prefix, and confirm it traverses the full R1 transport to a durable consume — then discard it before it can affect state.** The medium tier, per agent, executes **that agent's own provider-specific installed hook command** — the exact `command` string materialized into its config file (claude `settings.json`, agy/codex `hooks.json`), as located by the G4-Q3 shape table — with a distinguished synthetic event (`event = "selfcheck"`, `event_id = "selfcheck:{agent_id}:{boot_id}"`). **This must exercise each provider's actually-installed hook, not the claude shape as a stand-in (F-4):** the failure modes it catches (a `PATH` that lost `ah`, a broken sandbox mount, an unwritable outbox, an unarmed inotify) are per-agent and can differ by provider sandbox, so a claude-only probe would leave codex/agy path-rot undetected. ahd's consume path recognizes the `selfcheck:` prefix as a **liveness probe**: it verifies the outbox record was written and consumed (proving hook → outbox → inotify/scan → consume all work end-to-end) but routes it to a no-op sink instead of `mark_agent_idle_hook_event`, so a probe can never move `agents.state`.

**Rationale.** Config-presence (G4-Q3) proves the hook *string* is correct; it cannot prove the hook *runs* and its output *arrives*. A `PATH` that lost `ah`, a broken sandbox mount, an outbox directory that isn't writable, an inotify watch that was never armed — all leave the config byte-perfect while the wire carries nothing. Only pushing a real event through the real path detects those. The synthetic event must use the *same* transport as production (no test-only shortcut) or it proves nothing about production; the only difference is the terminal sink, gated on a reserved id prefix so the probe is inert by construction — never by a runtime flag that could be misconfigured. Running it against each provider's installed hook is what extends that proof from claude to all three wired providers.

- **North-star acceptance:** §五.4 (catches wiring rot the config check is blind to) + §五.5.
- **Dogfood node:** new ledger row **G4-synthetic-roundtrip** — break the outbox directory's write permission, run the medium check, assert it warns loudly and surfaces in `ah doctor` (and does *not* silently pass because the config string still looks right).

## G4-Q5: What does deep end-to-end on `ah doctor` add over the startup tiers?

**Decision: `ah doctor` runs the full matrix as operator-facing `DoctorCheck`s — including live-daemon consume confirmation and a dead-letter census — with the daemon unambiguously up and no auto-repair.** Deep mode adds `DoctorCheck` entries (`src/cli/doctor.rs`, alongside the existing `daemon_check`, `provider_health_checks`, etc.) covering: (a) synthetic round-trip *with confirmed ahd consume* (medium tier only confirms the transport up to consume; deep confirms against a definitely-serving daemon and reports the round-trip latency); (b) a census of every agent's `outbox/dead/` (R1-Q3) — any dead-letter is a `Warn` with the count and the first failure reason; (c) the cross-agent wiring matrix (which managed agents are fully wired vs. degraded). Each maps to a `DoctorStatus::{Pass,Warn,Fail}` with a `suggestion` (`src/cli/doctor.rs:15-19`), so it composes with the existing doctor output and `has_failures` gate (`:331`).

**Rationale.** The startup tiers optimize for *not slowing boot* and *self-healing what's cheap*; they deliberately don't do the expensive, definitely-daemon-up confirmations. `ah doctor` is where an operator asks "is the control path actually healthy right now," so it pays the full cost, reports everything, and — crucially — does *not* auto-repair (an operator running doctor wants a diagnosis, not silent mutation of their agents' configs mid-investigation). The dead-letter census on doctor closes the loop with R1-Q3: quarantined events are visible exactly where an operator looks for health.

- **North-star acceptance:** §五.4 ("深度端到端检查挂 `ah doctor`" is this section literally) + surfaces §五.1's dead-letter outcomes operationally.
- **Dogfood node:** **G4-synthetic-roundtrip** (doctor arm) + **R1-deadletter** (census arm — assert a quarantined event shows as a doctor `Warn`).

## G4-Q6: On detection, what is the "loud degradation + auto-repair" behavior — and when is repair unsafe?

**Decision: light-tier detection triggers (1) a loud structured log, (2) a self-check event on the same event spine, (3) an idempotent re-materialize repair, (4) a mandatory re-check that must pass before the agent is dispatch-eligible — and repair is *attempted*, never *assumed*.** The sequence on a light-tier failure:

1. **Loud log:** `tracing::error!` (not `warn!`) naming the agent, the specific unwired/drifted target (G4-Q3's presence pre-check gives the specificity), and the action being taken. "响亮" means error-level and specific, per north-star §二's degradation principle.
2. **Self-check event:** emit a `control_path_selfcheck` event through the *same* `events` channel the arbiter/transport use, so the incident is durable, auditable, and countable — a config that silently self-heals with no record would hide a real operational signal (someone or something is deleting hook configs). This event is telemetry, not a state transition.
3. **Auto-repair:** re-materialize the hook config via the shared `ah up` materialization path (G4-Q3), which is idempotent (it's the same NO_CHANGE-detecting path issue #13 hardened).
4. **Re-check gate:** immediately re-run the light-tier check; **only a passing re-check clears the dispatch block.** If repair fails (e.g., the settings file is read-only, the home is gone) the agent stays blocked, the log escalates, and it surfaces as a `Fail` in `ah doctor` — a fail-**closed** posture: an agent whose hook cannot be guaranteed wired is not dispatched to, because doing so would silently resurrect the fire-and-forget void.

**When repair is unsafe / must not be silent.** Auto-repair re-materializes *ah's own hook entries only*, per provider; it must not clobber operator or provider config it didn't author. Every provider's inject path already merges rather than overwrites — claude `materialize_claude_settings`, agy `merge_antigravity_hooks`/`inject_antigravity_hook_push`, **codex `merge_codex_hook_push`** — each via `remove_ah_owned_hook_groups` (drop only ah's prior entries) then re-push the current one (`home_layout.rs:419`, `:1180`, `:1193-1207`). G4 relies on that merge-preserving property; it adds no blind overwrite. This is **consistent with the F-3 narrowing, not in tension with it**: because the *drift check* now looks only at ah's own entries, and *repair* rewrites only ah's own entries, the detect→repair→re-check loop operates entirely within ah's namespace — an operator's out-of-band additions are neither flagged (F-3) nor touched (here), so the loop converges instead of oscillating. If drift is somehow detected in a way that repair *cannot* reconcile without discarding non-ah config, that is a `Fail` + escalate, not a forced overwrite. And repair is always *followed by verification* (step 4) — the north-star's §五.4 says "报警并自动修复", and an unverified repair is indistinguishable from no repair; the re-check is what makes "修复" a claim we can prove rather than assert.

**Rationale.** This is the north-star's §五.4 clause built as a closed loop: detect (G4-Q3) → alarm loud (1) → record (2) → fix (3) → prove-fixed-or-stay-blocked (4). The fail-closed dispatch gate is the load-bearing safety property: it is *better to not dispatch* than to dispatch to an agent that cannot report back, because a silent completion-signal loss is exactly the disease this whole spec exists to kill (handoff §域1: agy 假完成/假BUSY 占道). The self-check *event* (2) matters because a self-healing system that heals silently trains its operators to believe nothing is wrong while someone is, in fact, repeatedly deleting hook configs — the loud log is for the moment, the event is for the trend (dogfood ledger).

- **North-star acceptance:** §五.4 in full ("下次启动自检报警并自动修复,期间有响亮降级日志") + §五.5 (the whole loop is one required automated test: delete config → restart → assert alarm event emitted + config re-materialized + re-check passes + agent becomes dispatch-eligible only after).
- **Dogfood node:** **G4-wiring-selfheal** (full-loop arm) — this is the row that proves §五.4 end to end; its Gen-4+ observations feed the efficacy report (`research/gen-efficacy-reports.md`) with a治愈-实证 verdict only if the automated loop passes *and* a live-stack manual delete is observed self-healing.
