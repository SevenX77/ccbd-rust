# Per-Provider Completion-Gap Matrix — Divergence Draft (g1, blind/independent)

**Discipline of this document.** This is the *second* independent blind draft of the provider-completion-gap divergence (Track C step 2, `STAGE-PLAN-2026-07-11.md` §三). It was written **without reading `divergence-provider-matrix-o1.md`** — no comparison, no echo, no cross-reference to that draft, so that a later overlap analysis is meaningful. It lists **only questions, risks, attack surfaces, and failure modes**. It gives **no conclusions and no recommendations** — where a design choice is implied by a question, the choice is deliberately left open. Anti-sycophancy is explicit: the goal is to find holes in the "explicit-protocol-for-all-providers" premise, not to affirm it. The already-converged skeleton (explicit protocol as primary signal, pane-inference rejected, git-diff as an auxiliary job-gate not a primary signal — `perception-final-convergence-2026-07-09.md`) is **not** re-argued here; every question below assumes that skeleton and probes what it does *not yet* answer for heterogeneous providers.

**Grounding facts** (stated only to make the questions concrete, not as conclusions): `src/provider/manifest.rs:30` defines `enum CompletionSignalKind { LogOnly }` — a single variant, assigned to **every** provider (bash/codex/claude/antigravity, `:351/:381/:400/:418`). `src/provider/home_layout.rs:232` injects a `"Stop"` hook for claude and `:408 inject_antigravity_hook_push` injects a `"Stop"` hook for antigravity; **both** route to the identical `ah agent notify … --event stop` command (`build_ah_hook_command`, `:674`). The ah-side handler `handle_agent_notify` → `mark_agent_idle_hook_event` marks the agent idle; nothing in that path returns a `{"decision":"block"}` verdict back to the harness. These facts frame — but do not answer — the questions below.

## Axis A — agy without an enforceable Stop-block: what actually holds the turn?

- **A1.** The brief's premise is "agy 没有 Stop-hook 等价物," but the code injects an agy `"Stop"` hook (`inject_antigravity_hook_push`) that fires the same `--event stop` as claude. Is the real gap not "no Stop event" but "no *block* semantics on the Stop event"? I.e., does the antigravity harness (a) run the Stop hook *before* the turn is allowed to end and honor a non-zero exit / block verdict to hold it open, or (b) fire the Stop hook *after* the turn has already ended, as pure observe-with-no-veto? Where is that harness property documented, and how is it verified rather than assumed?
- **A2.** If agy's Stop hook is observe-only (no veto), then "强制声明" has no enforcement primitive for agy at all — the model literally cannot be prevented from ending. In that case, what does "explicit declaration protocol" even mean for agy operationally — is it reduced to a soft prompt instruction ("please call `ah job done`"), and if so, what is the failure mode when the model simply doesn't, and how does that differ from today's turn-end 假 COMPLETED?
- **A3.** Grounding: today *even claude's* `--event stop` is handled as an idle-marker, not as a block-with-reason gate — the enforcement half (R2's "Stop hook block+reason 强制未声明不许结束") is not present in the code path I can see. Is the "claude has a natural handle" advantage therefore *aspirational* (a capability R2 must still build) rather than *current*? If claude's own enforcement is unbuilt, does the claude-vs-agy asymmetry the matrix is premised on actually exist yet, or is it a future asymmetry?
- **A4.** Antigravity's hook timeout is 5000ms vs claude's 5ms (`hook_timeout_for_provider`, `:689`). What does that 1000× gap imply about agy's hook execution model — heavier startup? A different in-process vs out-of-process firing? Does a 5s hook budget interact badly with a Stop-*block* semantics if one existed (the turn stalls 5s on every end)? Does the timeout difference itself hint that agy's hook fires in a fundamentally different lifecycle position than claude's?
- **A5.** If agy has *no* block and the fallback is a watchdog on "process alive, no declaration, N seconds elapsed," what is N for agy — and how is N distinguished from a *legitimately* long agy task? agy's documented normal pattern is "end_turn, background task runs, self-wake later" (handoff §域1); under that pattern "process alive + no completion signal + long silence" is *normal*, not stuck. Does the same watchdog observable mean opposite things for claude (didn't declare = anomaly) and agy (backgrounded = normal)?
- **A6.** When a watchdog nudges agy "you didn't declare done," claude could (with block) re-prompt in the *same* turn; agy (no block) can only be re-dispatched as a *new* turn with fresh context. Is a "nudge to re-declare" even coherent for agy, or does the nudge land as a brand-new task that has lost the in-progress state? What prevents the nudge itself from being interpreted by agy as new work?
- **A7.** Does agy expose any hook point *other* than Stop (e.g. a tool-call hook, a pre-exit hook, a periodic heartbeat) that could carry a completion signal with different — possibly better — semantics than the Stop event? If such points exist, what is each one's firing guarantee and failure mode, and does relying on a non-Stop point create a per-provider special case?

## Axis B — trust of the declaration itself when the declarer is a known-unreliable narrator

- **B1.** If the completion protocol routes through agy self-reporting (`ah agent notify --event stop` or an explicit `ah job done`), and agy has a documented history of turn-end 假 COMPLETED (Gen-2 3/3, Gen-3 2例; handoff §域1), what structurally distinguishes a *true* explicit declaration from a *false* explicit declaration by the same unreliable narrator? Is a self-report from an untrustworthy source epistemically any better than a turn-end inference from that same source?
- **B2.** Does the design conflate two different trust boundaries: (i) "the signal was *delivered* reliably" (transport — R1's outbox solves this) vs (ii) "the signal is *true*" (semantic trust — unsolved)? R1 can guarantee a `stop` event arrives exactly once; it cannot guarantee the task was actually done. Which layer owns semantic truth, and is it under-specified precisely because R1's reliability makes the pipe *look* solved?
- **B3.** Claude's Stop-block (if built) enforces that the hook *fires*, but the *content* ("I completed the task") is still model-authored. Grounding: the Gen-1 ledger records claude itself hitting the semantic gap (job_e817301f: COMPLETED while background monitors still ran). So does the Stop hook catch "task actually done," or only "turn ended"? If a model end_turns with a background `cargo test` still running, does the block-unless-declared mechanism produce a *confident false declaration* rather than catching anything?
- **B4.** Anti-sycophancy: could *mandatory* declaration make 假完成 **worse** for a reflexive model? If agy (or claude) learns "I must emit a done signal to end my turn," does the forced ritual increase the rate of confident-but-false declarations rather than truth? Is there evidence either way, and what would measure it?
- **B5.** For corroboration, the git-diff evidence gate is blind to *non-mutating* tasks (审计, 设计发散, e2e verification) — exactly the class where job_e817301f 假完成 occurred. What corroborates a done-declaration for a task that legitimately produces zero filesystem mutation? Is the read-only-task class the permanent blind spot of the entire evidence-gate approach, and does that blind spot correlate with the highest 假完成 risk?
- **B6.** reply载荷错位 (#33: COMPLETED but reply_text was a brief fragment): if "task is done" and "here is the result payload" both ride self-report, can one be true while the other is garbage? What is the consumer's (master's) failure mode when `done=true` but `payload=garbage` — does it act on the garbage because the done flag was trusted? Should completion-truth and payload-fidelity be separate trust surfaces, and does treating them as one signal hide the #33 failure?
- **B7.** If a declaration must be cross-checked against an independent signal, which signals are *actually* independent of the model's self-report? OS liveness (T0) is independent but only answers "alive," not "done." Log-tail (T2) is derived from the same model's output. Git HEAD is independent of the model's *claim* but not of the model's *actions*. Is there any completion-corroborating signal that is genuinely orthogonal to the model, and if not, is "explicit protocol + corroboration" quietly circular?

## Axis C — protocolizing 产物轨 (git HEAD) into the FSM: which layer, and does it inherit pane-inference's sins?

- **C1.** Git HEAD change is currently a *human-master* manual anchor. If lifted into the arbiter FSM, which of T0–T3 does it occupy? It is a durable, re-readable filesystem artifact (a commit), so it is not transient like pane text — but is it T2 (persistent-artifact replay), or is it *outside* the T0–T3 agent-state hierarchy entirely because the arbiter design scoped git-diff as a *job-level gate*, not part of the agent state machine? Does the matrix need to resolve whether 产物轨 is an *agent-state* signal or a *job-gate* signal before it can be placed at all?
- **C2.** The brief asks whether git-evidence repeats pane-inference's error (不可重放/转瞬即逝). Git HEAD is the *opposite* on the replayability axis (durable, re-readable). But does it carry a *different* structural defect: HEAD-changed proves *a* commit happened, not that *this task* is complete or correct? What are the concrete false-positives (model commits a WIP checkpoint then keeps working → HEAD moved, task not done) and false-negatives (pure review/design/e2e task → HEAD unchanged, task IS done)?
- **C3.** Attribution: how does the FSM know *which* HEAD change belongs to *which* job? CLAUDE.md says the gatekeeper (writes tests) and the code-monkey (implements) **share a worktree serially** — so a commit's authorship-to-job mapping is ambiguous when two roles touch one tree. Is this the same class of attribution race as R1's hook-attribution problem, but for commits, and does it need the same per-attempt-cookie treatment?
- **C4.** Baseline capture: "HEAD changed since dispatch" requires a HEAD baseline snapshotted at dispatch time. Is that baseline captured atomically with dispatch? If dispatch and baseline-capture race, could a commit that landed *just before* dispatch be miscounted as this job's evidence, or a commit that lands *during* the race be missed?
- **C5.** Protocolizing 产物轨 makes the FSM depend on git I/O. What new failure modes does that import: git index-lock contention, detached HEAD, amended/squashed commits, submodules, worktree corruption, git-not-on-PATH? On the Windows-native target (handoff §域5) does the git-evidence path behave identically, or does it become a platform-conditional signal (present on Linux, degraded on Windows) — i.e., does 产物轨 itself become per-*platform* special-cased in addition to per-provider?
- **C6.** The evidence gate's two guardrails (is_mutating static annotation + third-attempt release) shift the trust to *whoever annotates is_mutating*. What is the failure when a mutating task is mis-annotated read-only (gate skipped → 假完成 slips through) vs a read-only task mis-annotated mutating (blocked until third-attempt release → latency + noise)? Is the annotation itself a new, unvalidated trust surface, and who/what validates it per dispatch?
- **C7.** If 产物轨 is protocolized, does it *only* fire for git-repo tasks? Many worker tasks (设计发散, spec drafting into .kiro, e2e verification) mutate non-git-tracked paths or nothing at all. Does a git-only evidence signal systematically under-serve exactly the non-code providers/roles, and does that skew the protocol toward code-implementation providers?

## Axis D — codex task_complete and the three-way unification without per-provider special-casing

- **D1.** Grounding: `CompletionSignalKind` today has one variant, `LogOnly`, for all providers — the manifest currently *flattens* the heterogeneity this matrix exists to model. Is `CompletionSignalKind` the intended seam where per-provider completion capability should be expressed (e.g. future variants `StopHookBlock` / `TurnEndObserve` / `TaskComplete`), or is it dead/vestigial? If the seam exists but is unused, what has been silently relying on all-providers-are-LogOnly, and what breaks when that flattening is removed?
- **D2.** What *is* codex's `task_complete` mechanically — a hook, an in-band tool call, an exit code, a structured output field? Does it fire at the same semantic point as claude Stop (turn boundary) or at "the agentic loop terminated" (a different F2/F3 boundary)? If codex's completion means "loop done" rather than "task done," does codex import the *same* F3≠F2 confusion as agy, just via a different primitive?
- **D3.** If the three primitives fire at three different semantic points (claude: turn-end-blockable; agy: turn-end-observe; codex: loop-terminate), can a single internal `job.done` event map all three *without* a `match provider { … }` in the completion logic? Grounding: provider-specific branching already exists (`hook_timeout_for_provider`, `materialize_antigravity_* vs materialize_claude_*`). Does the completion protocol inherit and *multiply* that branching, and where is the line between "acceptable adapter seam" and "per-provider special-case sprawl"?
- **D4.** Normalization discards nuance: if codex's task_complete carries a success/failure discriminant that agy's bare turn-end lacks, does normalizing all three to a lowest-common-denominator "done" event *lose* the failure signal? Is the common event schema a *union* (carries every provider's fields, mostly-null) or an *intersection* (only what all three share, losing the richest provider's information)? What breaks under each choice?
- **D5.** Version skew: claude's Stop-hook contract, agy's hooks.json shape, codex's task_complete are all *external* contracts that evolve under us (the repo already has `.SUPERSEDED` rule files evidencing churn). Does the protocol pin a per-provider contract version? What detects a provider *silently* changing its completion semantics — is there a G4-style self-check for the *semantic* contract, not just the wiring, and if not, how would a silent contract drift manifest (as a new 假完成 wave with no code change on our side)?
- **D6.** Capability matrix: what is the actual per-provider feature grid across (a) fires a completion signal, (b) can *block* until declaration, (c) carries a structured payload, (d) distinguishes success/failure, (e) can be re-entered after a nudge? Where the grid has holes, does the unified protocol degrade to the *weakest* provider's guarantees, target the *strongest*, or fork per-tier — and is any of those three explicitly chosen or silently defaulted?
- **D7.** Idempotency/conflict *within* one provider: if a provider emits more than one completion-relevant signal for the same job (e.g. agy turn-end Stop *and* an explicit `ah job done`), R1's event_id dedups the *transport*, but which signal wins *semantically*? What resolves a conflict where the Stop event says one thing and an explicit declaration says another for the same job?

## Axis E — additional structural differences and cross-cutting attack surfaces

- **E1.** Master as a fourth completion class: master (sonnet5) runs via `claude --continue` with `handle_master_notify` and master hook sentinels. The Gen-1 ledger states "模块C完成协议设计必须同时覆盖 master 等待唤醒" (master 裸等 = master's own F2≠F3). Is master a fourth provider-class with yet another completion semantics, and does the matrix cover master-self-completion or only worker completion? Does master's completion even *mean* the same thing (master orchestrates; its "done" is a different predicate than a worker's)?
- **E2.** The "suspended-pending-background" third state: agy's normal pattern (end_turn, background task, self-wake) is neither "done" nor "not-done" — it is "turn ended, task deliberately continuing off-turn." Does the protocol have a *state* for that, and which providers can even *express* it? If the protocol only has done/not-done, does every agy background task get mis-slotted into one of two wrong buckets?
- **E3.** Failure/crash vs finished-but-silent: claude's Stop hook fires on a clean end_turn — does it fire on *error exit / crash / OOM*? If a provider crashes mid-task, *no* completion signal fires (that's T0/F1 liveness territory). But the boundary between "crashed (T0 owns)" and "finished-but-didn't-declare (watchdog owns)" is provider-specific: for claude, alive + no-declaration = anomaly; for agy, alive + no-signal = maybe-normal-backgrounding. Does the *same* observable (alive, no completion signal, N seconds) require a per-provider interpretation, reintroducing the special-case smell?
- **E4.** Does a provider's completion hook even *fire* under the sandbox/credential model (模块D)? G4's wiring self-check (Track A) validates claude's `settings.json` hook path — does it equally validate agy's `.gemini/config/hooks.json` path and codex's path, or is the self-check itself claude-shaped? A provider whose hook can't reach ahd's socket from inside its sandbox is a per-provider *silent* completion loss that looks identical to 假完成.
- **E5.** Onboarding/first-turn: does completion behave differently on an agent's first turn (onboarding) vs steady state? agy onboarding zombies are a historical issue (handoff). Is there a window where the completion hook isn't wired yet but the agent can already accept a task, and does that window differ per provider?
- **E6.** Observability of the gap itself: the dogfood ledger counts 病种×代次 but attributes provider *manually*. How would we *measure* per-provider 假完成 rate to know if the unified protocol worked? Is there a per-provider completion-signal-fidelity metric, or does "did the protocol fix agy specifically" remain unfalsifiable from the data we collect?
- **E7.** The "wrap agy to synthesize a signal" temptation: if agy genuinely lacks an enforceable primitive, is there pressure to *synthesize* a completion signal for agy via a PTY/process shim — and would that quietly reintroduce the inference that pane-text was rejected for? Is there a spectrum between "native explicit protocol" and "synthesized/inferred," and does agy force a point on that spectrum closer to the rejected pane-inference than the converged skeleton admits? (Stated as an open risk, not a claim that it does.)
- **E8.** Does the protocol assume every provider's completion signal is *positive-only* (fires when done)? What about *negative* completion (task failed, model gives up, refuses)? Does each provider have a way to signal "I finished and I failed" distinct from "I didn't finish," and if agy/codex lack that, does every failure collapse into either false-done or watchdog-timeout — losing the model's own "I failed" testimony?
- **E9.** Concurrency/interleaving: if one agent is dispatched a second job while the first's completion signal is still in flight (dispatch-ACK race, handoff §域2), does the completion signal's *job attribution* survive across providers identically? Claude/agy/codex may buffer or coalesce turns differently — does a provider that coalesces two dispatches into one turn emit one completion signal for two jobs?
